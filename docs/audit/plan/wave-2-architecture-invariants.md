# Wave 2 — Architecture-invariant fixes — Remediation Plan

> Scope: restore the modular-hexagonal invariants the audit found breached, without changing behaviour. Ports stop leaking `anyhow` and return the typed `AgentError` (the `Sandbox` port and the config writers); `shared/infra/config` stops depending *into* `modules/tools`; the `sync` application service moves **all** its raw filesystem work (the `tokio::fs` calls *and* the inline `Path::exists()` reads) behind an infrastructure port like its `Git` sibling; the security-sensitive credential-resolution rule is single-sourced so `app.rs` and the TUI runtime can no longer drift (and the swallowed env-import persist failure becomes a visible Notice); the TUI runtime stops acting as a second composition root (the headless `kiri sync` route and the live `/sync` push both wire through `app.rs`, with the harness home single-sourced as a `Settings` field); and the one deliberate domain↔framework coupling (`InputBuffer` owning a `tui_textarea::TextArea`) is ratified by an ADR instead of left a silent breach. These ship as a unit because they are the dependency-graph corrections that Wave 3's god-file splits depend on: you cannot cleanly split `config.rs`, `runtime.rs`, or `view_state.rs` until their error types, imports, and composition are correct.

## Remediation steps (ordered to de-risk within the wave)

### Step 1 — Make the `Sandbox` port return `AgentError` (TOOL-01 / ARCH-01 / ARCH-02)
- **Problem** — The central filesystem-chokepoint port `Sandbox` declares `resolve_existing`/`resolve_create` with `anyhow::Result` (`application/sandbox.rs:3,34,38`), directly violating "ports return `AgentError`" and diverging from its sibling `CommandSandbox`, which correctly returns `Result<_, AgentError>` (`command_sandbox.rs:32-36`). The typed contract is lost at every tool boundary (each tool stringifies via `error.to_string()`), and a future `?`-propagating caller is broken.
- **Approach** —
  - In `src/modules/tools/application/sandbox.rs`: delete `use anyhow::Result;` and change the two fallible port methods to `fn resolve_existing(&self, rel: &str) -> Result<PathBuf, AgentError>;` and `fn resolve_create(&self, rel: &str) -> Result<CreateResolution, AgentError>;` (import `crate::shared::kernel::error::AgentError`).
  - Reuse the existing `AgentError::Sandbox(String)` variant for path-resolution refusals (no new variant — verified there are no exhaustive `match AgentError` sites). Broaden its doc-comment in `src/shared/kernel/error.rs:16-18` to: confinement-setup failures **and** path-resolution refusals (traversal, sensitive name, credential directory, not-found, escapes-root).
  - In `src/modules/tools/infrastructure/sandbox.rs`: convert the `impl Sandbox for FsSandbox` methods `resolve_existing`/`resolve_create` (and their private helpers `join_checked`, `assert_within`, `assert_not_sensitive`, `assert_not_in_secret_dir`) from `anyhow::Result`/`bail!`/`with_context` to `Result<_, AgentError>` returning `AgentError::Sandbox(...)`. Keep the existing message text. The inherent wiring constructors `with_confinement`, `new` (`#[cfg(test)]`), and `relocated` stay `anyhow::Result` — they are composition-root/binary-edge wiring, an allowance the audit (TOOL-01 recommendation) explicitly grants and which is clean because those constructors share no helpers with the port methods; keep `use anyhow::{Context, Result, bail}` only for those.
  - In `src/modules/tools/infrastructure/sensitive.rs`: `SensitiveMatcher::new` and `load_sensitive_matcher` are wiring constructors invoked only from the binary edge (after Step 2, `app::wire` is their sole caller). Leave them on `anyhow::Result` and note in the module doc that they are edge-only wiring. (This is the minimal closure of ARCH-02 consistent with the audit's stated edge allowance — no behaviour change.)
  - Tool call sites (`fs/read_file.rs:64`, etc.) keep `Err(error) => ToolOutcome::Error(error.to_string())` — `AgentError` is `Display`, so they compile unchanged.
- **Files touched** — `src/modules/tools/application/sandbox.rs` (mod), `src/modules/tools/infrastructure/sandbox.rs` (mod), `src/shared/kernel/error.rs` (doc only), `src/modules/tools/infrastructure/sensitive.rs` (doc only).
- **Tests first** — In `infrastructure/sandbox.rs` tests, before the change, add error-*type* assertions (the existing `.is_err()` tests only prove failure, not the typed contract):
  - `resolve_existing_traversal_yields_agenterror_sandbox` — `matches!(sb.resolve_existing("../etc"), Err(AgentError::Sandbox(_)))`.
  - `resolve_create_sensitive_yields_agenterror_sandbox` — `..Err(AgentError::Sandbox(_))` for a `.env` create through a guarded sandbox.
  - `resolve_existing_credential_dir_yields_agenterror_sandbox` — `.ssh/config` refusal carries `AgentError::Sandbox`.
  - `resolve_existing_missing_yields_agenterror_sandbox` — a non-existent path returns `AgentError::Sandbox`, not a swallowed string.
  - Regression lock: `sandbox_port_error_type_is_agenterror` — a compile-asserting binding `let _: fn(&FsSandbox, &str) -> Result<PathBuf, AgentError> = FsSandbox::resolve_existing;` (or simply that the existing happy-path tests still pass with the new signature).
- **ADR** — none (conforms to the existing ADR 0003 invariant; this is a fix toward it).
- **Risk & dependencies** — Blast radius is the `tools` module plus the one-line `AgentError::Sandbox` doc. Low: callers stringify, tests assert `.is_err()`. No cross-wave dependency. Done first because it is the headline High and the central port everything else reads through.

### Step 2 — Stop `shared/infra/config` depending into `modules/tools` (SHARED-01 / ARCH-04)
- **Problem** — `config.rs:11-12` imports `NetworkPolicy` (a tools *application* enum) and `SensitiveMatcher` + `load_sensitive_matcher` (a tools *infrastructure* adapter + its constructor). `shared` is the leaf every module depends on; here a foundation file reaches *down* into one module's adapter wiring, and it is what propagates ARCH-02's `anyhow` into `Settings::resolve`.
- **Approach** —
  - **SensitiveMatcher:** Remove the `pub sensitive: SensitiveMatcher` field from `Settings` (`config.rs:612`) and the `sensitive: load_sensitive_matcher()?` initializer (`config.rs:722`), and delete the `use crate::modules::tools::infrastructure::sensitive::…` import (`config.rs:12`). In `src/app.rs::wire`, replace `settings.sensitive.clone()` (`app.rs:74`) with a locally built matcher: `let sensitive = load_sensitive_matcher()?;` (import it from `crate::modules::tools::infrastructure::sensitive`) and pass `sensitive` into `FsSandbox::with_confinement`. The env-read + default list stay single-sourced in `tools/infrastructure/sensitive.rs`; the composition root (which is *allowed* to know every module) does the wiring. `anyhow` on `load_sensitive_matcher()` is fine here — `wire` returns `anyhow::Result` at the binary edge.
  - **NetworkPolicy:** Relocate the pure-data enum `NetworkPolicy { Deny, Allow }` from `src/modules/tools/application/command_sandbox.rs` into `shared/kernel` (a new `src/shared/kernel/sandbox.rs`, re-exported from `shared/kernel.rs`), as a cross-cutting sandbox primitive beside `ApprovalMode`. Update `command_sandbox.rs`, `application/sandbox.rs`, `infrastructure/sandbox.rs`, `infrastructure/exec.rs`, `infrastructure/confine/macos.rs`, and the nine `fs/*` tools to import it from `shared::kernel::sandbox` (mechanical import-path change across the 15 files the audit enumerated; `SandboxPolicy` stays in `command_sandbox.rs` and now imports `NetworkPolicy` from kernel). `config.rs` keeps its `sandbox_network: NetworkPolicy` field but imports it from `shared::kernel` — no longer a `shared → module` edge. Delete the `use crate::modules::tools::application::command_sandbox::NetworkPolicy` import (`config.rs:11`).
  - After this step `config.rs` has zero `use crate::modules::…` lines (assert with a grep in the DoD).
- **Files touched** — `src/shared/infra/config.rs` (mod), `src/shared/kernel/sandbox.rs` (new), `src/shared/kernel.rs` (mod: declare+re-export), `src/app.rs` (mod), `src/modules/tools/application/command_sandbox.rs` + `application/sandbox.rs` + `infrastructure/{sandbox,exec}.rs` + `infrastructure/confine/macos.rs` + `infrastructure/fs/{read_file,write_file,delete_file,delete_dir,create_dir,list_dir,move_path,search,run_command}.rs` (import paths only).
- **Tests first** —
  - `config_has_no_module_imports` — a repo guard test asserting `src/shared/infra/config.rs` contains no `use crate::modules::`. Encode as a `#[test]` that reads the file via `include_str!` and asserts `!contents.contains("use crate::modules::")`.
  - Reuse the existing sandbox tests: they still build `FsSandbox` with a `SensitiveMatcher` directly, so the relocation of the matcher build to `wire` must not regress them (they pass unchanged).
  - `wire_builds_sensitive_matcher` — extend the existing wiring/characterization coverage to assert a guarded sandbox still refuses `.env` after `wire` (proves the matcher is injected, not dropped).
- **ADR** — none (the relocation is small and conforms to ADR 0003's "shared is the leaf"; note the move in the file's module doc).
- **Risk & dependencies** — The NetworkPolicy relocation is wide but purely mechanical (import paths); the matcher removal is local. No behaviour change. Independent of Step 1 but cleaner after it. Touches `config.rs`, which Wave 3 (SHARED-03) will split — do it here first.

### Step 3 — Config writers return `AgentError`, not `anyhow` (SHARED-02)
- **Problem** — `persist_active_model`/`persist_effort`/`persist_active_provider`/`upsert_provider` (and their shared `update_global_config`) return `anyhow::Result<()>` (`config.rs:363,378,390,395,404`) yet are invoked deep in the live `/models`/`/effort`/`/provider` effect handlers (`runtime.rs:977,1015,1053,1127`) — `anyhow::Error` flowing back into runtime-reachable code, against the invariant.
- **Approach** —
  - Add a new variant to `src/shared/kernel/error.rs`: `#[error("config error: {0}")] Config(String)` with a doc-comment: "A config read/encode/write failure while persisting a live `/models`/`/effort`/`/provider` change." (Distinct from `Io` because the TOML encode failure is not an `io::Error`, and distinct from `Sandbox`/`Secret`.)
  - Change `update_global_config` and the four public writers to `-> Result<(), AgentError>`. Inside `update_global_config`, map each fallible step explicitly: `read_config_file(path).map_err(|e| AgentError::Config(e.to_string()))?`, `toml::to_string_pretty(&config).map_err(|e| AgentError::Config(format!("failed to encode config: {e}")))?`, `std::fs::create_dir_all(parent).map_err(|e| AgentError::Config(...))?`, `std::fs::write(...).map_err(|e| AgentError::Config(...))`. Keep `read_config_file`, `validate_config_str`, and `Settings::resolve` on `anyhow::Result` — they run only at boot (the binary edge), which the audit affirms is defensible.
  - The runtime call sites already `if let Err(error) = config::persist_…(…) { push Notice }` — the type change is transparent; no runtime edit needed beyond the format specifier already using `{error:#}` (works for `AgentError`). The four call sites use `.unwrap()` in `config.rs` tests (config.rs:1028-1039) and need no change.
- **Files touched** — `src/shared/kernel/error.rs` (add variant), `src/shared/infra/config.rs` (writer signatures + explicit maps).
- **Tests first** — In `config.rs` tests:
  - `persist_active_model_returns_agenterror_config_on_unwritable_path` — point `config_path` at a path whose parent cannot be created (e.g. under an existing *file*) and assert `matches!(persist_active_model(bad, "nvidia", "m"), Err(AgentError::Config(_)))`.
  - `persist_effort_round_trips_and_is_ok` — happy path writes and re-reads `effort`.
  - `upsert_provider_adds_then_persist_active_provider_chains` — the runtime's `.and_then` chain (`runtime.rs:1127`) compiles and succeeds with the new error type.
  - Regression lock: `config_writers_do_not_return_anyhow` — `include_str!` guard asserting the four `pub fn persist_*`/`upsert_provider` signatures contain `Result<(), AgentError>` (or simply a binding `let _: fn(&Path, Effort) -> Result<(), AgentError> = config::persist_effort;`).
- **ADR** — none.
- **Risk & dependencies** — Local to `config.rs` + one error variant. Runtime call sites unchanged. Independent; sequenced after Step 2 only to keep all `config.rs` edits contiguous before Wave 3's split.

### Step 4 — Single-source credential resolution (ARCH-05 / ROOT-02 / ERR-02)
- **Problem** — The keyless→stored→env-import-with-best-effort-persist rule is implemented twice: `app::resolve_credential` (`app.rs:390-424`, returns `Ok(None)` for onboarding, persist reported via `eprintln!`, env-import **gated** on `AuthMethod::ApiKey`) and `ProviderSwap::resolve_credential` (`runtime.rs:144-167`, returns `Err` for missing, persist swallowed with a bare `let _ =` at `runtime.rs:160`, env-import **not** gated). Two copies of a security-sensitive rule guarantee drift; they *already* diverge (ERR-02: one surfaces the persist outcome, one swallows it; and only one gates env-import on the auth method).
- **Approach** —
  - Add a single resolver in `src/modules/provider/infrastructure/factory.rs` (next to `api_key_from_env`, which it needs): `pub fn resolve_credential(profile: &ProviderProfile, secrets: &dyn SecretStore) -> Result<CredentialResolution, AgentError>` returning a small enum that single-sources the *policy* while leaving *reporting* and *absent-handling* to each caller:
    ```
    pub enum CredentialResolution {
        Keyless,                                   // auth = none → Credential::None
        Stored(Credential),                        // found in the secret store
        Imported { credential: Credential,         // one-time env-var import
                   persisted: Result<(), AgentError> },
        Absent,                                    // nothing configured
    }
    ```
    The resolver owns the order (keyless short-circuit → `secrets.get` → `api_key_from_env` gated on `AuthMethod::ApiKey` → `Absent`) and performs the best-effort `secrets.set`, returning its `Result` in `Imported.persisted` so **both** callers can surface it (closing ERR-02). The env-import gate adopts the stricter `AuthMethod::ApiKey` check (the runtime copy omitted it — adopting the gate is the safer convergence).
  - *Layering note (justification):* the resolver lives in `provider/infrastructure` because the env/keyring access it orchestrates (`api_key_from_env`, `SecretStore`) is a binary-edge credential-acquisition concern, not domain policy; co-locating it with `api_key_from_env` avoids inventing an application port for a helper only the composition root and the live-swap call. This is a deliberate, documented placement, recorded in one line in the function's doc-comment.
  - `app::resolve_credential` becomes a thin adapter over it: `Keyless => Ok(Some(Credential::None))`, `Stored(c)/Imported{credential,..} => Ok(Some(credential))`, `Absent => Ok(None)`; on `Imported`, report `persisted` via the existing `eprintln!` (boot edge).
  - **ERR-02 surfacing (concrete):** `ProviderSwap::resolve_credential` returns the `Credential` *and* the persist outcome. Replace the current 2-tuple return of `switch_to` (`runtime.rs:192-204`) with a small struct so the persist warning reaches the UI:
    ```
    struct ProviderSwitch {
        provider: Arc<dyn CompletionProvider>,
        model: String,
        persist_warning: Option<AgentError>,   // Some when an env-import key failed to persist
    }
    ```
    `switch_to` maps the resolution: `Keyless => Credential::None` (no warning), `Stored => credential` (no warning), `Imported{credential, persisted}` → `credential` with `persist_warning = persisted.err()`, `Absent => Err(AgentError::Provider("no credential for provider '…'"))`. `apply_set_provider` (`runtime.rs:1039`) destructures the struct: push the success Notice as today, then `if let Some(warning) = persist_warning { push an Error Notice }`. This replaces the swallowed `let _ =` at `runtime.rs:160`. `add_and_activate` (`runtime.rs:209-231`) is a separate path that already propagates its `secrets.set` via `?` and returns its own 2-tuple to `apply_save_provider` — it needs **no** change.
- **Files touched** — `src/modules/provider/infrastructure/factory.rs` (new resolver + enum), `src/app.rs` (delegate), `src/modules/tui/infrastructure/runtime.rs` (delegate; `ProviderSwitch` struct; surface persist outcome in `apply_set_provider`).
- **Tests first** — In `factory.rs` tests (reuse the `FakeStore` pattern already in `app.rs` tests):
  - `resolves_keyless_to_keyless_resolution` — `auth = None` → `CredentialResolution::Keyless`, never consults the store (assert with a store double that panics on `get`).
  - `resolves_stored_credential` — store returns a key → `Stored`.
  - `resolves_env_import_and_reports_persist_ok` — no stored key, env var set, store `set` ok → `Imported { persisted: Ok(()) }`.
  - `resolves_env_import_surfaces_persist_failure` — store `set` errors → `Imported { persisted: Err(_) }` (regression lock for ERR-02: the failure is *returned*, never swallowed).
  - `resolves_absent_when_nothing_configured` — no store, no env → `Absent`.
  - `env_import_is_gated_on_api_key_auth` — a non-ApiKey auth does not import from env.
  - In `runtime.rs` tests: `switch_to_propagates_persist_warning` — a `ProviderSwap` over a store whose `set` errors, switched to a provider whose key lives only in env, yields `persist_warning: Some(_)` (regression lock that `apply_set_provider` can render it).
  - Keep `app.rs`'s existing `resolve_credential_*` tests green (they now exercise the delegate); keep the existing `switch_to_*` runtime tests green against the new `ProviderSwitch` return.
- **ADR** — none.
- **Risk & dependencies** — Moderate: touches boot and the live `/provider` switch, both security-sensitive. The enum + `ProviderSwitch` struct make the convergence explicit and testable. No cross-wave dependency. Do after Steps 1-3 so the resolver lands on already-clean error types.

### Step 5 — Move *all* `SyncService` filesystem I/O behind an adapter (ARCH-03 / SYNC-06)
- **Problem** — `SyncService` (application layer) abstracts git behind the `Git` port and NDJSON behind `memory_ndjson` (infrastructure), but does its work-tree/config filesystem work inline — both with `tokio::fs` (`sync_service.rs:61,68,127,136,149,200-205` + the private `write_atomic` at `:239-248`) **and** with raw std `Path::exists()` reads (`dir.join(".git").exists()` at `:62` in `init` and `:209` in `require_initialized`; `self.config_path.exists()` at `:201` in `export_profile`; `incoming_config.exists()` at `:126` in `pull`). A split-personality layer: git and NDJSON are behind adapters, the third dependency is raw I/O in the use-case. `sync` is an allowed fs owner, so this is a layer-purity inconsistency, not an invariant *breach* — but it diverges from the module's own pattern, and the `.exists()` reads are easy to miss (they are not `tokio::fs`).
- **Approach** —
  - Add a port in `src/modules/sync/application/work_tree.rs`: `#[async_trait] pub trait SyncWorkTree: Send + Sync` with capability-named methods returning `Result<_, AgentError>`:
    - `ensure_dir(&self, dir: &Path) -> Result<()>` — replaces `create_dir_all` (`:61`).
    - `write(&self, path: &Path, contents: &str) -> Result<()>` — replaces `fs::write` for `.gitignore` (`:68`, `:200`).
    - `copy(&self, from: &Path, to: &Path) -> Result<()>` — replaces `fs::copy` (`:202`).
    - `read_to_string(&self, path: &Path) -> Result<Option<String>>` — `None` = not found, so the caller's NotFound-vs-other distinction is preserved. This single method **eliminates two of the `.exists()` reads**: at `:126-127` the `incoming_config.exists()` + `read_to_string` pair collapses to a `match work_tree.read_to_string(&incoming_config).await? { Some(text) => …, None => "no config in sync" }`; at `:136-140` the baseline read keeps its three-way (Some / empty-on-NotFound / None-on-other) shape, encoded inside the adapter and surfaced as the existing `Option`-plus-error contract (the adapter returns `Ok(Some(""))` semantics are kept by the caller mapping a genuine NotFound here — see note).
    - `write_atomic(&self, path: &Path, contents: &str) -> Result<()>` — replaces `:149` (and the free `write_atomic` at `:239-248`).
    - `exists(&self, path: &Path) -> Result<bool>` — replaces the remaining `.exists()` reads: `dir.join(".git").exists()` at `:62` (init) and `:209` (`require_initialized`), and `self.config_path.exists()` at `:201` (export, the copy-if-present guard). One general method covers all three; the `.git`-presence semantics read clearly at the call site (`if !self.work_tree.exists(&dir.join(".git")).await?`).
    - *Baseline-read note:* the `pull` baseline at `:136-140` distinguishes Ok(text) / NotFound→`Some(String::new())` / other-error→`None`. Encode this exactly with `read_to_string` returning `Result<Option<String>>` where the adapter maps `ErrorKind::NotFound` to `Ok(None)` and the caller treats `Ok(None)` for *this* path as the empty baseline; any non-NotFound io error propagates as `Err(AgentError::Sync(..))` and the caller's existing `None`/unreadable branch is driven by catching that. (Keep the current observable behaviour: a missing current config = empty baseline; an unreadable one = require `--force`.)
  - Add the adapter in `src/modules/sync/infrastructure/fs_work_tree.rs`: `pub struct FsSyncWorkTree;` implementing the port with `tokio::fs`, including the temp-then-rename `write_atomic` lifted verbatim from `sync_service.rs:239-248` (keeping the `.{name}.kiri-tmp` convention). Map `io::Error` → `AgentError::Sync(...)`.
  - `SyncService` gains a `work_tree: &'a dyn SyncWorkTree` field (mirroring `git: &'a dyn Git` and `memory: &'a dyn SharedMemory`). Replace every inline `fs::…` and `.exists()` call in `init`/`push`/`pull`/`export_profile`/`require_initialized` with `self.work_tree.…`. Drop `use tokio::fs;` and the free `write_atomic` from the application file. The `TrustView`/`risky_config_changes` block is unchanged (SYNC-04, its relocation to `domain/`, is a different wave — leave it).
  - Callers (`app::wire_sync` headless and the runtime's `sync_push` via the injected `SyncContext`, both reworked in Step 6) supply an `FsSyncWorkTree`; the runtime never constructs it (it comes from `wire` in `SyncContext`).
- **Files touched** — `src/modules/sync/application/work_tree.rs` (new), `src/modules/sync/infrastructure/fs_work_tree.rs` (new), `src/modules/sync.rs` + `application/mod.rs` + `infrastructure/mod.rs` (declarations), `src/modules/sync/application/sync_service.rs` (consume the port; remove inline fs + `.exists()`; add `work_tree` field + `new` param).
- **Tests first** —
  - The existing `SyncService` tests use a real `FakeGit` over `TempDir`; they now also pass a real `FsSyncWorkTree`. Keep them green (init/push/pull/force/safe/schema-invalid) — they are the behavioural regression lock that the adapter extraction changed nothing.
  - `fs_work_tree_write_atomic_replaces_target` — adapter unit test: write `a`, then `write_atomic` `b`, assert content is `b` and no `.tmp` sibling remains.
  - `fs_work_tree_read_to_string_returns_none_when_missing` — locks the NotFound→`None` mapping the `pull` baseline + the `incoming_config` collapse depend on.
  - `fs_work_tree_exists_reports_presence` — `exists` returns true/false for a created/absent path (locks the `.git`/config-present checks).
  - `sync_service_has_no_inline_fs` — `include_str!` guard asserting the **non-test** portion of `sync_service.rs` (the slice before `#[cfg(test)]`) contains none of `tokio::fs`, `use tokio::fs`, `.exists()`, `std::fs`, or `fs::`. (Scoping to the pre-test slice is required because the test module legitimately uses `std::fs` over `TempDir`.)
- **ADR** — none (conforms to ADR 0015 + ADR 0003; the module already follows this pattern for `Git`).
- **Risk & dependencies** — Contained to the `sync` module. The new port is the home for the later SYNC-05 `write_atomic` de-dup (other wave). Sequenced before Step 6 because Step 6 injects an `FsSyncWorkTree` through `SyncContext`.

### Step 6 — Collapse the TUI runtime's second composition root (ARCH-06 / ROOT-01 / TUII-03)
- **Problem** — `app::wire` is the documented single composition root, but adapter construction has leaked into two other places. `main.rs::run_sync` (`main.rs:29-45`) builds `SqliteSharedMemory` + `GitCli` + `SyncService` directly and hardcodes `global_dir.join("memory").join("shared.db")` — a literal duplicate of `Settings::shared_memory_db` (`config.rs:742`); if that default moves, `kiri sync` silently operates on the wrong DB. The runtime's `sync_push` (`runtime.rs:1367-1390`) does the same: it derives `global_dir` from `config_path.parent()`, recomputes the shared-DB path, and `new`s `SqliteSharedMemory`/`GitCli` inside the front-end. There is also no single source for the harness home: `global_dir` exists only as a local in `Settings::resolve` (`config.rs:674`), so every consumer re-derives it (the `config_path.parent()` magic this wave is meant to eliminate).
- **Approach** —
  - **Single-source the harness home (prerequisite):** add an explicit `pub global_dir: PathBuf` field to `Settings` (`config.rs:605-653`), populated in `resolve` from the already-computed `kiri_global_dir()` (`config.rs:674`) — `global_dir: global_dir.clone()` in the struct literal. Every sync consumer now reads `settings.global_dir`; no `config_path.parent()` derivation survives.
  - **Headless route (ROOT-01):** Add `pub async fn wire_sync(settings: &Settings, action: SyncAction) -> anyhow::Result<()>` to `src/app.rs`. It reads `settings.global_dir`, `settings.shared_memory_db`, and `settings.config_path` (single source), builds `SqliteSharedMemory` (+`init`), `GitCli`, `FsSyncWorkTree`, and `SyncService`, runs the action, and prints the summary. `main.rs` shrinks to: `let settings = Settings::resolve(cli.path, cli.prompt)?;` then `if let Some(CliCommand::Sync { action }) = cli.command { return app::wire_sync(&settings, action).await; }` — entry-only dispatch, no adapter construction, no hand-rebuilt paths. (`Settings::resolve` is already side-effect-light and TTY-independent, so resolving it before the sync branch is safe; `kiri_global_dir`/`SyncService` imports leave `main.rs`.)
  - **Live `/sync` route (TUII-03 / ARCH-06) — decision: inject ports, no adapter choice in the front-end.** Define a `SyncContext` built entirely in `wire` and injected into `Tui::new`:
    ```
    struct SyncContext {
        git: Arc<dyn Git>,
        memory: Arc<dyn SharedMemory>,
        work_tree: Arc<dyn SyncWorkTree>,
        global_dir: PathBuf,
        config_path: PathBuf,
    }
    ```
    All three adapters (`GitCli`, `SqliteSharedMemory`, `FsSyncWorkTree`) are constructed **in `wire`** behind their ports; the paths come from `settings.{global_dir,config_path}`. `wire` builds the sync `SharedMemory` via a small `build_sync_memory(settings) -> Arc<dyn SharedMemory>` helper that opens `settings.shared_memory_db` with best-effort `init` (mirroring `build_memory`'s non-fatal-init pattern at `app.rs:212-225`); it is a second SQLite handle to the same file as the memory tools, which is safe and is exactly what the prior on-demand open did transiently. `sync_push` then constructs **no adapter and recomputes no path** — it only assembles the use-case: `SyncService::new(ctx.git.as_ref(), ctx.global_dir.clone(), ctx.config_path.clone(), ctx.memory.as_ref(), ctx.work_tree.as_ref())` and calls `.push()`. The `use …SqliteSharedMemory` / `…GitCli` imports leave `runtime.rs` entirely; the adapter *choice* lives only in `wire`. (This is the concrete resolution of the finding: the front-end stops being a composition root. The one accepted consequence — wire opening `shared.db` even when `KIRI_MEMORY=off` — is recorded in Open Questions.)
  - This also resolves the network-I/O grep noise the audit noted: the bare `reqwest::Client` already lives in `ProviderSwap` for live provider rebuilds (legitimate, never sends here) — leave it; the finding's structural concern is the *sync* path, addressed above.
- **Files touched** — `src/main.rs` (shrink to dispatch; drop sync/memory imports), `src/app.rs` (`global_dir` read, `wire_sync`, `build_sync_memory`, build/inject `SyncContext`), `src/shared/infra/config.rs` (`Settings::global_dir` field + initializer), `src/modules/tui/infrastructure/runtime.rs` (`Tui`/`SyncContext` hold the ports; `sync_push` uses them; drop `SqliteSharedMemory`/`GitCli` imports and the hardcoded path), `src/modules/tui/...` (`Tui::new` signature).
- **Tests first** —
  - `wire_sync_uses_settings_shared_memory_db` — drive `wire_sync` with a `Settings` whose `shared_memory_db`/`global_dir` point at a temp dir, run `SyncAction::Status` on an uninitialized tree, and assert the "not initialized" error path — proving it never reconstructs the path by hand. Regression lock for ROOT-01's stale-DB drift.
  - `settings_exposes_global_dir` — assert `Settings::resolve` populates `global_dir` and that `shared_memory_db`/`config_path` are children of it (locks the single source).
  - `runtime_has_no_sync_adapter_construction` — `include_str!` guard asserting `runtime.rs` contains no `SqliteSharedMemory`, no `GitCli`, and no `join("memory").join("shared.db")` literal (proves the adapter choice + path left the front-end).
  - Keep the existing `SyncService` behaviour tests (Step 5) green — the composition move changes wiring, not behaviour.
- **ADR** — none (re-affirms ADR 0003's single-composition-root; record the `SyncContext` injection and the `build_sync_memory` decision in `app.rs`'s module doc).
- **Risk & dependencies** — Depends on Step 5 (injects `FsSyncWorkTree`) and benefits from Step 4 (clean credential path nearby). Touches `runtime.rs`, which Wave 3 (TUII-01) will split — doing it here first means the split carves an already-clean composition. The `Settings::global_dir` field lands here but edits `config.rs`, kept logically grouped with the other config edits (Steps 2-3) that precede Wave 3's `config.rs` split.

### Step 7 — Ratify the domain-owned input widget by ADR (TUIC-01)
- **Problem** — `src/modules/tui/domain/view_state.rs:1-2` imports `ratatui::style::Style` and `tui_textarea::{TextArea,…}`, and `InputBuffer` embeds a stateful `TextArea<'static>` (cursor, viewport, soft-wrap, undo/redo) and exposes `set_styles`/`widget` over `ratatui` types — the *only* domain file in the codebase importing UI-framework crates, breaching "domain = pure data, no frameworks" (ADR 0003:28). It is a *deliberate, documented* tradeoff (the widget is the editor's single source of truth) but it is implicit; the contract requires deviations to be ratified, not silent.
- **Approach** — Ratify, do not relocate (relocating the `TextArea` behind a pure-data shadow would duplicate cursor/wrap/undo state and is the heavier, lower-value path). Write `docs/decisions/0017-tui-input-buffer-domain-widget.md` (verified: 0017 is the next free number — 0001-0016 exist):
  - **Decision (one line):** `InputBuffer` may own a `tui_textarea::TextArea` inside `tui/domain` as the editor's authoritative state; this single file is the *only* sanctioned domain↔UI-crate coupling, it exposes pure accessors, and all clipboard/render side effects stay in `tui/infrastructure`.
  - Context: the Elm reducer needs the editor's cursor/selection/undo state to be the model's source of truth; reimplementing a soft-wrapping multi-line editor as pure data is speculative re-work with no present benefit; the widget is already firewalled (one file, accessor-only).
  - Consequences + guardrail: no *other* domain file may import a UI crate; record this as an enforced boundary. Update ADR 0003's invariant note (and `CLAUDE.md`'s domain description if needed) to cross-reference 0017 as the one sanctioned exception.
  - Add a guard test (in a crate-level test module): `only_input_buffer_couples_domain_to_ui_crates` — walk **all** `*.rs` recursively under every `src/modules/*/domain/` directory (not just the top level — a future nested `domain/<sub>/foo.rs` must not silently re-breach), and assert none except `tui/domain/view_state.rs` (or its Wave-3 successor `input_buffer.rs`) contains `use ratatui` / `use tui_textarea`. Implement the recursive walk with a small `std::fs::read_dir` recursion (or `walkdir` if already a dev-dependency — verify before adding). This makes the ADR's boundary enforceable so a future domain file cannot silently re-breach it.
- **Files touched** — `docs/decisions/0017-tui-input-buffer-domain-widget.md` (new), `docs/decisions/0003-modular-hexagonal-architecture.md` (cross-reference note), optionally `CLAUDE.md` (one-line exception note), one guard test file.
- **Tests first** — The guard test above is the contract: `only_input_buffer_couples_domain_to_ui_crates` (written before the ADR is "done"), plus it doubles as the regression lock that no new domain file imports `ratatui`/`tui_textarea`.
- **ADR** — **ADR 0017** (see Approach). This is the wave's one architectural ratification.
- **Risk & dependencies** — Documentation + one guard test; no production code change. Independent; last because it is the only non-code step and gates nothing.

## Open Questions
- **Eager-vs-lazy shared store for sync (resolved, with a flagged consequence).** Step 6 injects an `Arc<dyn SharedMemory>` built once in `wire` (`build_sync_memory`), removing the `SqliteSharedMemory` *choice* from the front-end — this is the decision. The residual, accepted consequence: when memory is disabled (`KIRI_MEMORY=off`) `wire` still opens `settings.shared_memory_db` for sync, which creates an empty `~/.kiri/memory/shared.db` even for a user who never runs `/sync`. Accepted because sync is orthogonal to the memory-tool feature flag (it owns the shared store as its own data source) and the cost is one cheap SQLite open of an already-owned data dir. The considered alternative — a `Fn`-based sync-service factory injected from `wire` to keep the open lazy — was rejected as speculative abstraction (YAGNI) for a single call site. Flagged here so a reviewer can object to the empty-file side effect before implementation; if rejected, the fallback is the lazy factory.
