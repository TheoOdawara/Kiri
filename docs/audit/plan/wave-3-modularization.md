# Wave 3 — Modularization of oversized files — Remediation Plan

> Scope: dissolve every oversized, multi-responsibility file in the crate into small, single-purpose modules **without changing behavior** — `runtime.rs` (2072 raw / ~1627 prod, the only true god-file), `Tui::run` (~410-line effect loop, runtime.rs:301-687), `config.rs` (1054 raw / ~827 prod, five+ concerns), `keymap.rs` (2274 raw / ~946 prod), `view_state.rs` (767 raw grab-bag), `markdown.rs` (654 raw / ~521 prod), `AgentLoop::run` (247 lines, four levels deep), the sync trust DTOs, and the oversized inline `#[cfg(test)]` modules that distort the size census. Each change is a pure move + re-wire (or, for `AgentLoop::run` and `Tui::run`, a structural split fully wrapped by the existing test suite), named for its one responsibility, with every existing test still green. It ships as a unit because it is one coherent theme — file/folder structure — with no functional surface change, and it lands **after** waves 1–2 have cleaned the cross-context dependencies and duplication these files carry, so each god-file is opened once.

## Remediation steps (ordered to de-risk within the wave)

### Step 1 — Extract oversized inline test modules to sibling files (STRUCT-07)
- **Problem** — A raw `wc -l` census is misleading: `agent_loop.rs` is 1431 lines but only ~360 prod (1071 inline test lines), `keymap.rs` 2274/~946, `registry.rs` ~636/~94, `view.rs` ~416/~85, `sandbox.rs` ~663/~313. The giant inline `mod tests` forces a reader to scroll past a thousand test lines and makes these files *look* like the worst offenders. Maintainability only — no behavior.
- **Approach** — For each file, replace the inline `#[cfg(test)] mod tests { … }` with a declaration `#[cfg(test)] mod tests;` and move the body verbatim to a sibling file. Privacy is preserved because the child `tests` module still sees the parent's private items via `use super::*`.
  - **`agent_loop.rs` and `keymap.rs`** (the two files this wave later promotes to a directory — Steps 7 and 6): declare `#[cfg(test)] mod tests;` directly and create `agent_loop/tests.rs` / `keymap/tests.rs`. In a non-`mod.rs` file, `mod tests;` already resolves to `<name>/tests.rs`, and the `<name>.rs` file legally coexists with the sibling `<name>/` directory (the crate already relies on this `<name>.rs` + `<name>/` convention, e.g. `infrastructure.rs` + `infrastructure/`). **No temporary `#[path]` hedge is needed** — the directory-bound resolution works the moment the directory exists, independent of whether the production split (Step 6/7) has run.
  - **`registry.rs`, `view.rs`, `sandbox.rs`** (stay single-file, no directory): use `#[path = "<name>_tests.rs"] #[cfg(test)] mod tests;` so no directory is created for an otherwise-unsplit file. `#[path]` is reserved for exactly these three.
  - In `sandbox.rs`, only the `#[cfg(test)] mod tests` at line 351 moves; the `#[cfg(test)] use … NoConfinement` (line 10) and the `#[cfg(test)] pub fn new` (line 44) are production-file test fixtures and **stay in `sandbox.rs`** (the moved test module reaches `FsSandbox::new` via `super`).
  - In `registry.rs` move the block at line 96; in `view.rs` the block at line 86.
- **Files touched** — Modified: `src/modules/agent/application/agent_loop.rs`, `src/modules/tui/application/keymap.rs`, `src/modules/tools/application/registry.rs`, `src/modules/tui/infrastructure/view.rs`, `src/modules/tools/infrastructure/sandbox.rs`. New: `…/agent_loop/tests.rs`, `…/keymap/tests.rs`, `…/registry_tests.rs`, `…/view_tests.rs`, `…/sandbox_tests.rs`.
- **Tests first** — This *is* the test move; there is no new contract. The regression lock is mechanical: capture the green `cargo test` baseline (test count + names) **before** the move; after the move assert the identical set of test functions runs (`cargo test -- --list` diff is empty) and all pass. No test body byte changes beyond an added `use super::*;` where the inline module relied on the enclosing scope.
- **ADR** — none.
- **Risk & dependencies** — Very low (pure relocation). No cross-wave dependency. De-risks Steps 6 and 7 by shrinking those files to their real prod surface first.

### Step 2 — Move sync trust policy into the domain (STRUCT-12 / SYNC-04)
- **Problem** — `sync_service.rs` carries a self-contained, pure (no-I/O) trust sub-feature — `TrustView`/`TrustProvider`/`TrustSandbox`/`TrustEmbeddings` (lines 254–289) + `risky_config_changes` (292–367) — alongside the git/fs `SyncService` orchestration. Its sibling pure rule `incoming_wins` correctly lives in `sync/domain/merge.rs`; this placement is inconsistent and burdens the service with two responsibilities (orchestration **and** the security trust policy), making the gate easy to overlook when reasoning about the security surface.
- **Approach** — Create `src/modules/sync/domain/config_trust.rs`. Move the four `Trust*` structs and `risky_config_changes` verbatim. `risky_config_changes` becomes `pub(crate) fn risky_config_changes(current: &str, incoming: &str) -> Vec<String>`; the `Trust*` structs stay module-private. This is consistent with `merge.rs`, which already does pure `time`-crate parsing in `domain/` — pure `toml::from_str` parsing (no I/O) is the same precedent. Declare `pub mod config_trust;` in the existing `src/modules/sync/domain/mod.rs` (keep its current `mod.rs` style; the convention unification is STRUCT-06, a different wave). In `sync_service.rs`, replace the moved code with `use crate::modules::sync::domain::config_trust::risky_config_changes;` (the call site at line 142 keeps working).
- **Test move (surgical, NOT a verbatim block move)** — `sync_service.rs:369` holds a **single** `#[cfg(test)] mod tests` that mixes the trust-policy unit tests (the `risky_config_changes` cases at lines 470–561) with the `SyncService` orchestration tests (which import `SqliteSharedMemory` and `GitOutput`, lines 580+). `sync_service.rs` is **not** in the STRUCT-07 list, so its test module stays inline. Therefore: extract **only** the `risky_config_changes` test functions (470–561) into a **new** `#[cfg(test)] mod tests` inside `config_trust.rs` (with its own `use super::*;`); leave the `SyncService` tests in `sync_service.rs`'s inline `mod tests`. After the move, `sync_service.rs`'s test module no longer references `risky_config_changes` directly (those tests are gone), so no import is needed there for the trust fn.
- **Files touched** — New: `src/modules/sync/domain/config_trust.rs` (production + its own trust-test module). Modified: `src/modules/sync/application/sync_service.rs` (remove trust code + the trust tests; keep the SyncService tests), `src/modules/sync/domain/mod.rs`.
- **Tests first** — The moved trust tests (`risky_config_changes` happy/edge/regression cases, incl. `unreadable_current_config_is_treated_as_risky` at 560) are the regression lock for the policy and must pass unchanged in their new home. The retained `SyncService` tests (`pull_refuses_a_risky_config_without_force`, `pull_applies_a_risky_config_with_force`, …) must still compile and pass in `sync_service.rs`. No new behavior, no new cases.
- **ADR** — none.
- **Risk & dependencies** — Low; single-module, isolated. This step touches **only** placement, not the gate's known logic holes (SYNC-01 `require → os`, SYNC-02 magic strings) — those are separate security findings owned by another wave; do not fold their fixes in here, but landing this move first makes that wave's edit land in `domain/config_trust.rs`.

### Step 3 — Split `markdown.rs` along the `Block` AST boundary (STRUCT-10)
- **Problem** — ~521 prod lines hold two phases of an inline markdown engine: the **parse** layer (`Block`, `InlineAccum`, `FmtTag`, `BlockBuilder`, `ParseCtx` + its impl) and the **render-to-ratatui-lines** layer (`render_block`, `wrap_spans`, `prepend_prefix`). They separate cleanly at the `Block` type.
- **Approach** — Promote to a `markdown/` submodule behind the existing facade `markdown.rs` (`<name>.rs` + `<name>/` convention; `markdown.rs` declaring `mod parse;` resolves to `markdown/parse.rs`).
  - `markdown/parse.rs`: `Block`, `InlineAccum`, `FmtTag`, `BlockBuilder`, `ParseCtx` and its impl. Export `Block` and a single entry `pub(super) fn parse(markdown: &str, base: Style) -> Vec<Block>` that runs the `Parser::new_ext` event loop currently inlined in `render_uncached`. `Block` is declared `pub(super)` so the sibling `render` module can match on it.
  - `markdown/render.rs`: `render_block`, `wrap_spans`, `prepend_prefix`, exporting `pub(super) fn render_blocks(blocks: &[Block], width: usize) -> Vec<Line<'static>>`. It `use super::parse::Block;`.
  - `markdown.rs` facade keeps the public `render` + `render_uncached`, `MAX_RENDER_CACHE`, `CacheKey`, and the `RENDER_CACHE` thread-local. `render_uncached` becomes: `let blocks = parse::parse(markdown, base); render::render_blocks(&blocks, width.max(1))`. Public API `markdown::render` is unchanged, so `transcript_pane`/approval call sites are untouched.
- **Files touched** — New: `src/modules/tui/infrastructure/markdown/parse.rs`, `…/markdown/render.rs`. Modified: `src/modules/tui/infrastructure/markdown.rs` (now a facade). `src/modules/tui/infrastructure.rs` (`pub mod markdown;`) unchanged.
- **Tests first** — Keep the existing `markdown` tests in the facade `markdown.rs` (they exercise `render`, the public seam, so they validate parse+render together — the strongest characterization). Add no new behavior. Regression lock: `renders_bold_and_italic_as_styled_spans`, `wraps_long_lines_preserving_styles`, `fenced_code_block_fills_the_width_with_background`, `blank_line_count_matches_raw_wrap_for_scroll_stability`, `render_is_memoized_and_stays_consistent` must pass unchanged. (If TUII-07's dead `&mut Vec<Block>` param on `start` is fixed in an earlier wave, the `parse` signature reflects that; otherwise leave the behavior identical.)
- **ADR** — none.
- **Risk & dependencies** — Low; self-contained module, well-tested at the public seam. Best landed after TUII-04 (word-wrap dedup) so `wrap_spans` is touched once.

### Step 7 — Split `AgentLoop::run` into `decide_and_run` + `checkpoint_transition` helpers (AGENT-01)
- **Problem** — `run` (lines 111–358) is a single ~247-line function nesting `loop → for call → match mode → match io.decide` up to four levels deep, carrying five responsibilities (round streaming, plan-mode short-circuit, within-round runaway checkpoint, per-mode per-call decision, result/abort bookkeeping). The per-mode decision block alone (~229–306) is ~77 lines that independently call `io.tool_started`/`timed`. It is the most-touched function in the module and its depth makes it risky to change.
- **Approach** — Extract two private async/sync methods on `AgentLoop`, keeping behavior byte-identical:
  - `async fn decide_and_run<IO>(&self, sandbox, call: &ToolCall, command: &str, mode: &mut ApprovalMode, io: &mut IO) -> Option<(ToolOutcome, Duration)>` — the body of the `match mode { Auto | Plan(blocked) | Plan | Default }` decision tree. It owns `io.tool_started` and `timed`, mutates `mode` on `ApprovedAuto`, and returns `None` on user abort (signalling the caller to answer-and-end). This collapses the deepest nesting out of `run`.
  - `fn checkpoint_transition(&self, decision: Approval, checkpoint: &mut Instant, calls_since_checkpoint: &mut usize, mode: &mut ApprovalMode) -> CheckpointFlow` where `enum CheckpointFlow { Continue, End(TurnOutcome) }` — the shared `Approved => reset` / `ApprovedAuto => switch-to-Auto + reset` arms used by both the within-round and post-round checkpoints; the call sites keep only their distinct `Declined`/`Aborted` end-of-turn handling (the within-round case must `answer_unanswered` first). This also realizes the AGENT-04 dedup (coordinate so it is not done twice).
  - `run` then reads as: `stream round → finish_turn → plan check → for each call { if cap reached: checkpoint_transition; let command = …; decide_and_run; record/answer } → post-round checkpoint_transition`. The `schemas`-fixed-for-the-turn invariant and every `answer_unanswered` early-exit are preserved exactly.
- **Files touched** — Modified: `src/modules/agent/application/agent_loop.rs` (prod only; tests already at `agent_loop/tests.rs` from Step 1).
- **Tests first** — This is a structural split of behavioral code, so the **existing** test suite is the contract and must stay green with no edits — it already covers every branch this split touches. Must-pass regression locks: `auto_mode_confirms_destructive_delete`, `auto_mode_confirms_out_of_root_target`, `the_call_cap_pauses_within_a_single_oversized_round`, `provider_failure_propagates_after_finishing_the_render`, plus the plan-mode block, declined-checkpoint answer-all, and abort answer-all cases. Before refactoring, confirm the full suite is green; after each extraction, re-run `cargo test agent_loop` and require zero diff in pass/fail set. Add no new tests (no new behavior) — the value is structural only.
- **ADR** — none.

### Step 8 — Split `runtime.rs` into a `runtime/` submodule with a `RunLoop` state struct + context structs (STRUCT-01 / TUII-01 / TUII-05 / BUILD-04 / BUILD-06)

### Step 9 — Extract `Tui::run`'s effect dispatch into `RunLoop::apply_effect` (STRUCT-02)
- **Problem** — `run` (301–687) spans ~410 lines; its `for effect { match effect { … } }` (424–686) is an 18-arm dispatch where several arms inline 30–60 lines of business logic (`SubmitPrompt`, `ResumeLast`, `OpenSession`, `ChangeWorkspace`, `NewSession`, `ApprovePlan`) while four others already delegate to `apply_*`. The inconsistent depth obscures the loop's control flow and makes `run` untestable as a unit.
- **Approach** — Done together with Step 8 (same file), built on the `RunLoop` state struct.
  - **Signature (corrected).** `runtime.rs` declares `use anyhow::Result;`, and the inline arms call `drive_turn(...).await?` (lines 445/628) and `draw_and_copy(...)?` (399) — so the extracted dispatcher MUST return a `Result`, not a bare value: `async fn apply_effect(&mut self, effect: Effect, ui: &mut UiDriver<'_>, engine: &mut EngineHandles<'_>) -> Result<()>` (i.e. `anyhow::Result<()>`, the crate's edge type already imported here). The `?` operators inside the moved arms propagate unchanged.
  - **No `ControlFlow`.** The outer loop is `while !model.should_quit { … }`; the only termination is `Effect::Quit => self.model.should_quit = true` (and the events-stream `None`/`Command::Quit` cases that also set `should_quit`). No arm `break`s the loop. A `ControlFlow<()>` return would therefore be dead — `run` loops on `self.model.should_quit` and never inspects a control-flow return value. So `apply_effect` returns `Result<()>`; loop exit stays governed solely by `should_quit`.
  - **Per-arm `&mut self` handlers.** Each currently-inline arm (`SubmitPrompt`, `ResumeLast`, `OpenSession`, `ChangeWorkspace`, `NewSession`, `ApprovePlan`) becomes a named `pub(super)` `&mut self` method on `RunLoop` in the relevant area module (`turn.rs`/`session_ops.rs`/`distill.rs`), taking only the effect payload + `&mut UiDriver` + `&mut EngineHandles`. Because the owned run state lives on `self`, each handler stays ≤ 7 params with **no new `#[allow]`** — this is the concrete answer to the arg-count gap. `apply_effect` is then a thin `match effect { … => self.submit_prompt(text, images, ui, engine).await?, … }` over the 18 arms.
  - `run` reduces to: stamp render → `select!` one input (events / ticker) → `for effect in update(&mut self.model, msg) { self.apply_effect(effect, &mut ui, &mut engine).await?; }` → check `self.model.should_quit` → loop, with the final teardown distillation (`self.drive_distillation(&mut ui)`) unchanged.
- **Files touched** — Modified: `src/modules/tui/infrastructure/runtime.rs` (facade `run` + `apply_effect`) and the area modules from Step 8 that receive the per-arm `RunLoop` methods.
- **Tests first** — `run` drives real terminal/engine I/O, so it stays covered by the pure `update`/`on_turn_end` unit tests (already moved) plus the run-to-verify gate. The regression lock is behavioral parity: the seeded-prompt path, `/new` distillation, `/resume`, `/sessions` open, `/cd` re-key, and plan approval must behave identically — assert via the existing `update`/`keymap` tests that produce these effects, and via the manual launch verification below.
- **ADR** — none.
- **Risk & dependencies** — Paired with Step 8 (same blast radius). Sequenced last.
