# Wave 1 — Security & correctness — Remediation Plan

> Scope: close every security and correctness gap that lets a hostile repo, a synced config, or a misbehaving/compromised provider read secrets, weaken the sandbox, OOM the process, or silently corrupt recall. This wave ships first because every later refactor wave inherits these surfaces; landing the typed sandbox/auth gate, the single-sourced credential-dir policy, the streamed-byte caps, the model-scoped recall, and the timed/atomic I/O here de-risks all of them. No feature work — hardening and correctness only.

## Remediation steps (ordered to de-risk within the wave)

### Step 1 — Single-source the credential-directory policy (SEC-03, TOOL-04)
- **Problem** — `SECRET_DIRS` (`sandbox.rs:19`) and `SECRET_HOME_DIRS` (`confine/macos.rs:13`) are byte-identical lists backing two enforcement layers (path-resolution refusal + Seatbelt read-deny) with nothing tying them together; a future edit to one silently weakens the other. This duplication is the root cause that makes SEC-02 easy to introduce and hard to notice.
- **Approach** — Add a new infrastructure module `src/modules/tools/infrastructure/secret_paths.rs` owning the single source of truth:
  - `pub(crate) const SECRET_DIRS: &[&str]` (the credential dir names, moved verbatim from `sandbox.rs`).
  - `pub(crate) const HOME_SECRET_FILES: &[&str]` — well-known home credential files for the OS deny layer: `.netrc`, `.npmrc`, `.pypirc`, `.pgpass`, `.git-credentials` (these mirror names already in `DEFAULT_SENSITIVE_PATTERNS`).
  - `pub(crate) const HARNESS_PRIVATE_DIR: &str = ".kiri"` — the harness's own credential/state dir, denied to confined children.
  Register the module in `tools/infrastructure.rs`. **Critically:** `sandbox.rs` keeps the path `sandbox::SECRET_DIRS` resolvable for its existing importers — `search.rs:18` does `use ...::sandbox::SECRET_DIRS;` and `run_command.rs` reuses it — by **re-exporting** rather than re-importing privately: delete the local `const` in `sandbox.rs` and add `pub(crate) use crate::modules::tools::infrastructure::secret_paths::SECRET_DIRS;`. `secret_dir_component`, `search`'s `--exclude-dir`, and `run_command` all keep compiling unchanged (no caller import churn). Keep the doc comment explaining why dir-level guarding is needed (the file-name guard matches files, not dirs). No behavior change yet — this is the structural prerequisite.
- **Files touched** — new `src/modules/tools/infrastructure/secret_paths.rs`; modified `src/modules/tools/infrastructure.rs`, `src/modules/tools/infrastructure/sandbox.rs`.
- **Tests first** — `secret_paths_dirs_match_resolution_guard`: assert `secret_dir_component` flags every name in `SECRET_DIRS` (lock the wiring). Existing `secret_dir_component_*`, `resolve_refuses_*` (sandbox.rs), and the `search` exclude-dir tests continue to pass against the re-exported const (regression lock — confirms the `pub(crate) use` path is intact).
- **ADR** — none (refactor only; the boundary ADR is updated in Step 3).
- **Risk & dependencies** — Low blast radius (a `use` re-export swap). Prerequisite for Step 2 and Step 3.

### Step 2 — Extend the macOS Seatbelt read-deny to Kiri's own store and home credentials (SEC-02)
- **Problem** — The Seatbelt profile denies only `SECRET_HOME_DIRS` reads; it does not deny `~/.kiri` (which holds `credentials.json`, the `0600` API-key fallback at `config.rs:744`) nor common home credential files, so a confined `run_command` can `cat ~/.kiri/credentials.json` or `~/.netrc` and return the bytes to the model. The file-tool guard already blocks `credentials.json`, so the omission is an inconsistency.
- **Approach** — In `confine/macos.rs::build_profile`, after the existing `SECRET_DIRS` loop (now imported from `secret_paths`, replacing the deleted `SECRET_HOME_DIRS`):
  - emit `push_deny_read(home.join(HARNESS_PRIVATE_DIR))` so `~/.kiri` is read-denied for confined children;
  - emit `push_deny_read(home.join(file))` for each `HOME_SECRET_FILES` entry.
  Keep the existing `extra_ro` re-allow emitted last so `KIRI_SANDBOX_RO_PATHS` can still punch a legitimate read-hole. Update the module doc-comment to name the now-covered set. Do **not** tighten the `(allow default)` base in this wave (flagged as an open question).
- **Files touched** — `src/modules/tools/infrastructure/confine/macos.rs`.
- **Tests first** — extend `profile_confines_writes_and_denies_secret_reads`: assert the profile contains `file-read*` denies for `.kiri`, `.netrc`, and `.git-credentials`. New `profile_denies_kiri_credential_store`: build a profile with a fake HOME and assert the `~/.kiri` deny rule is present. Regression: the existing `.ssh` assertion still holds.
- **ADR** — covered by Step 3's ADR update (the deny-set is part of the boundary).
- **Risk & dependencies** — Low (macOS-only profile text); depends on Step 1. No effect on non-macOS (NoConfinement) — that residual is addressed by the boundary in Step 3.

### Step 3 — Ratify and harden the run_command secrets boundary (SEC-01)
- **Problem** — File tools route every path through the sensitive-name + secret-dir chokepoint, but `run_command` validates only its `cwd`; the command body is arbitrary shell, so `cat .env` / `cat ~/.aws/credentials` returns exactly the material the guard exists to block. The only compensating control is the OS sandbox, which is macOS-only (NoConfinement elsewhere). This routes around the entire secrets-chokepoint investment.
- **Approach** — Three layers, none claiming a guarantee a shell parser cannot give:
  1. **OS layer** — Steps 1–2 make the macOS confinement deny-read the same credential set the file-tool guard blocks (the realistic v1-target path).
  2. **Defense-in-depth heuristic (explicitly non-guaranteeing)** — add a free function `references_sensitive_path(command: &str, sandbox: &dyn Sandbox) -> Option<String>` (in `run_command.rs`) that whitespace-tokenizes the command and returns the first token whose basename matches `sandbox.is_sensitive_name(..)` or whose path components hit `SECRET_DIRS` (reuse `is_sensitive_name` + a `secret_paths` helper). In `RunCommand::confirmation`, when it returns `Some(name)`, prepend a loud warning to the existing pt-BR prompt (e.g. `ATENÇÃO: este comando referencia um caminho sensível ('{name}'). …`). The default stays decline (`confirm(action, false)`), so this only makes an already-confirmed action scarier — never silently allows, and never silently denies. Document in the function's doc-comment that this is best-effort only (obfuscation — `c''at $E""NV`, base64, indirect reads — trivially evades token-matching) and the OS sandbox is the real control; it must not be sold as a guarantee (security theater is a defect). Dropping it entirely is a live open question.
  3. **Documentation** — update ADR 0009 to state the boundary explicitly: run_command's secret protection is the OS confinement layer (now covering the credential set) plus the always-on confirmation; on platforms without OS confinement run_command is not secret-guarded and the user-confirmation is the sole control.
- **Files touched** — `src/modules/tools/infrastructure/fs/run_command.rs`; `docs/decisions/0009-os-level-command-sandbox.md`; optionally one sentence in the system prompt (`config.rs SYSTEM_PROMPT`) noting run_command is shell-level and outside the file-tool secret guard.
- **Tests first** — `references_sensitive_path_flags_env_and_secret_dir`: `cat .env` → `Some(".env")`, `cat ~/.aws/credentials` → `Some` (matches `.aws`), `cargo build` → `None`. `confirmation_warns_on_sensitive_reference`: the confirmation prompt for `cat .env` contains the warning and still has `default_accept == false`. Regression: `run_command_defaults_to_decline` unchanged.
- **ADR** — Update `0009-os-level-command-sandbox.md`: "run_command secret boundary = OS confinement (credential-set read-deny, single-sourced) + mandatory confirmation; unconfined platforms rely on confirmation alone; the command-text heuristic is best-effort UX, not a control."
- **Risk & dependencies** — Medium readability/UX surface; depends on Steps 1–2.

### Step 4 — Cap and bound the provider stream surfaces (PROV-01, PROV-06)
- **Problem** — The OpenAI SSE path bounds the running total of streamed bytes by `MAX_STREAM_BYTES` (a forever-streaming endpoint would otherwise OOM the process, since `read_timeout` only bounds idle time between chunks). The Anthropic accumulator has no equivalent ceiling. Separately, in-band SSE error text (`{"error": …}` on a 200 stream) reaches the transcript uncapped, unlike HTTP error bodies which go through `truncate_body`.
- **Approach** —
  - **Single-source the constant:** move `MAX_STREAM_BYTES` from `openai/sse.rs` to a `pub(crate)` const in `provider/infrastructure.rs` (the module root) so both adapters reference one value. Update `openai/sse.rs` to use it.
  - **PROV-01:** add `streamed_bytes: usize` to the Anthropic `TurnAccumulator`; in `apply_delta`, before pushing a `TextDelta` into `content` or an `InputJsonDelta` into a tool-use slot, `saturating_add` the delta length and return `Err(AgentError::Provider("provider stream exceeded the maximum response size …"))` when it crosses `MAX_STREAM_BYTES` — mirroring the openai check at `openai/sse.rs:46-60`.
  - **PROV-06:** extract the bounded-preview logic from `http_error.rs` into `pub(crate) fn bounded_preview(text: &str) -> String` (keeping `MAX_ERROR_BODY_CHARS` as the single cap). **Keep the existing private `truncate_body(body: String) -> String` as a thin wrapper** (`bounded_preview(&body)`) so the three existing `http_error.rs` tests (`truncate_body_keeps_short_bodies_verbatim`, `truncate_body_caps_long_bodies`, and the import at line 52) compile and pass unchanged. Route `openai/sse.rs::format_stream_error`'s `message` and `anthropic/sse.rs`'s `Error` arm (`error.message`) through `bounded_preview`.
- **Files touched** — `src/modules/provider/infrastructure.rs`, `openai/sse.rs`, `anthropic/sse.rs`, `http_error.rs`.
- **Tests first** — `anthropic_stream_exceeding_cap_fails` (feed >`MAX_STREAM_BYTES` of text deltas → `Err`), and the parallel for tool-input deltas. `anthropic_normal_turn_under_cap_ok` (regression: a normal turn still assembles). `in_band_error_message_is_bounded`: an oversized in-band error message is truncated with the `… (truncated)` marker in both adapters. Keep `error_event_fails_the_turn`/`in_band_error_event_is_surfaced_as_an_error` and all three `truncate_body*` tests green.
- **ADR** — none.
- **Risk & dependencies** — Low; independent of the sandbox steps. Provides the shared `MAX_STREAM_BYTES`/`bounded_preview` that the later PROV-02 dedup wave should consume.

### Step 5 — Make embedding alignment and the env key boundary correct (PROV-04, PROV-08)
- **Problem** — `align_embeddings` checks count only, then stable-sorts by `index`; a response with duplicate/incomplete indices (two rows `index:0`) passes the count check and returns input 0's vector for two inputs — exactly the recall corruption the guard claims to prevent. Separately, `api_key_from_env` returns a plain `String`, so the env key lives as a non-zeroized heap copy until the caller wraps it.
- **Approach** —
  - **PROV-04:** after `data.sort_by_key(|d| d.index)`, verify `data.iter().enumerate().all(|(i, d)| d.index == i)`; on any gap/duplicate return `AgentError::Provider("embeddings response has non-contiguous indices …")`. Then the doc-comment's guarantee holds.
  - **PROV-08:** change `api_key_from_env` to `Option<Secret>` (wrap with `Secret::new` at the `std::env::var` site). Update callers `app.rs:404` (drop the `Secret::new(key)` re-wrap — it already gets a `Secret`) and `runtime.rs:154`.
- **Files touched** — `src/modules/provider/infrastructure/openai/embeddings.rs`, `factory.rs`; `src/app.rs`; `src/modules/tui/infrastructure/runtime.rs`.
- **Tests first** — `align_rejects_duplicate_indices` (two rows `index:0`, expected 2 → `Err`), `align_rejects_a_gap` (`index:0` + `index:2`, expected 3 → `Err`); keep `align_reorders_by_index_and_checks_count` and `align_rejects_a_count_mismatch`. `api_key_from_env_wraps_in_secret`: set a `KIRI_<ID>_API_KEY`, assert the returned `Secret` exposes the value and its `Debug` is redacted.
- **ADR** — none.
- **Risk & dependencies** — Low; PROV-08 touches two call sites (verify build after the signature change).

### Step 6 — Type the sync trust gate and close the sandbox-downgrade hole (SYNC-01, SYNC-02)
- **Problem** — `risky_config_changes` flags the sandbox only when the incoming mode is exactly `"off"` (`sync_service.rs:349-353`), missing a `require → os` downgrade (a genuine weakening on a platform with no OS sandbox). The gate's trust-view fields are `Option<String>` (`TrustProvider.auth`, `TrustSandbox.mode/network`, lines 266-285) compared against hand-typed literals (`"none"`, `"off"`, `"allow"`) decoupled from `AuthMethod`/the sandbox enums, so a variant rename leaves the magic strings silently mis-detecting.
- **Approach** —
  - **Typed primitives in the kernel (architecture-correct placement):** add a new `src/shared/kernel/sandbox.rs` (registered in `shared/kernel.rs`) holding:
    - `pub enum SandboxMode { Off, Os, Require }` with `pub fn rank(&self) -> u8` (`Off=0, Os=1, Require=2`) and a hand-written forward-compat `Deserialize` mapping **any unrecognized value to `Os`** (matching `resolve_sandbox_mode`'s default), plus `pub fn from_config(raw: Option<&str>) -> SandboxMode` with **`None` → `Os`** and unknown → `Os`. Document the ranking `Require > Os > Off`, so only a *strictly-lower* incoming rank flags.
    - `pub enum NetworkStance { Deny, Allow }` with a forward-compat `Deserialize` (unknown → `Deny`) and `from_config(None) = Deny`.
    These live in `shared/kernel` (alongside `AuthMethod`/`ProviderKind`) precisely so the future `sync/domain` gate (SYNC-04) can depend on them without `domain → infra`/`domain → tools` violations.
  - **config.rs derives runtime values from the kernel primitives (single source of parse):** refactor `resolve_sandbox_mode` to build `SandboxMode::from_config(config.or(env))` then map to `(enabled, require)` (removing the inline magic-string match); refactor `resolve_sandbox_network` to map `NetworkStance::from_config(...)` to the tools `NetworkPolicy` runtime enum. config keeps owning the env precedence; the *parse* now has one home.
  - **Trust-view field type change (the compile-correct fix):** change `TrustProvider.auth` to `Option<AuthMethod>`, `TrustSandbox.mode` to `Option<SandboxMode>`, and `TrustSandbox.network` to `Option<NetworkStance>` (all `#[serde(default)]`; each enum's forward-compat `Deserialize` keeps unknown values non-fatal). In `risky_config_changes`:
    - auth-disabled: flag when `cur.auth != Some(AuthMethod::None) && inc.auth == Some(AuthMethod::None)` (absent still treated as keyed — `None != Some(None)`).
    - sandbox downgrade: compute `current_rank = current.mode.map(|m| m.rank()).unwrap_or(SandboxMode::Os.rank())` and likewise for incoming (absent → `Os` baseline), and flag when `incoming_rank < current_rank` (covers `require→os`, `require→off`, `os→off`; `os→require` does not flag).
    - network widened: flag when `incoming.network == Some(NetworkStance::Allow)` and `current.network.unwrap_or(NetworkStance::Deny) != NetworkStance::Allow`.
    Import `AuthMethod` from `shared::kernel::provider` and the new enums from `shared::kernel::sandbox` (the gate no longer needs `shared::infra::config` for these comparisons).
- **Files touched** — new `src/shared/kernel/sandbox.rs`; modified `src/shared/kernel.rs`, `src/shared/infra/config.rs`, `src/modules/sync/application/sync_service.rs`.
- **Tests first** — `detects_sandbox_require_to_os_downgrade` (the audited hole), plus `require_to_off`, `os_to_off`; `os_to_require_is_safe` and `absent_to_os_is_safe` (None baseline = Os, no flag). `auth_gate_uses_typed_authmethod`: assert `risky_config_changes` flags an `api-key → none` auth change and that the comparison is the typed `Some(AuthMethod::None)` (the magic string `"none"` no longer appears in the gate — a `grep` lock). Kernel unit tests: `sandbox_mode_from_config_none_and_unknown_are_os`, `sandbox_mode_rank_orders_require_os_off`, `network_stance_from_config_defaults_deny`. Keep all existing `detects_*`/`*_is_safe` trust tests green (regression).
- **ADR** — `0015-portable-profile-sync.md`: note the trust gate now ranks sandbox modes and reasons over the typed kernel `AuthMethod`/`NetworkStance`/`SandboxMode` (no magic strings).
- **Risk & dependencies** — Medium (security control). The kernel placement of `SandboxMode`/`NetworkStance` is the cross-wave enabler for SYNC-04's move of the gate to `sync/domain` (keeps it pure).

### Step 7 — Bound and fail-close the sync import/merge/transport (SYNC-03, SYNC-08, SYNC-09, SYNC-11)
- **Problem** — `import` slurps the whole untrusted `memory.ndjson` with `read_to_string` (line 54) before `IMPORT_CAP` applies, so a multi-gigabyte remote file OOMs before the cap is consulted. `incoming_wins` falls back to the known-wrong lexicographic compare on parse failure (merge.rs:11-19), letting a crafted `updated_at` ("zzzz") overwrite a local entry. `git remote add origin <url>` forwards an unvalidated URL (an `ext::sh -c …` remote is a code-exec vector). The exported `memory.ndjson` (personal/preference content) is written world-readable `0644` (line 40).
- **Approach** —
  - **SYNC-03:** in `memory_ndjson::import`, add `const MAX_IMPORT_BYTES: u64` (a documented ceiling well above any real personal memory); stat the file first (`tokio::fs::metadata`) and return `AgentError::Memory("sync memory file too large …")` if it exceeds the cap. Replace `fs::read_to_string` + `content.lines()` with a streamed `tokio::io::BufReader::new(File).lines()` loop, keeping the `IMPORT_CAP` entry count. Update the comment to state the real guarantee (byte cap + entry cap).
  - **SYNC-08:** in `merge.rs::incoming_wins`, change the `_ =>` arm from `incoming_updated_at > existing_updated_at` to `false` (fail closed — a non-comparable timestamp keeps the existing entry). Update the doc-comment to say so.
  - **SYNC-09/SEC-04:** add `fn validate_remote_url(url: &str) -> Result<()>` in `sync_service.rs`, called at the top of `init`: accept `https://`, `http://`, `ssh://`, `git@…:` (scp-like), and an existing local filesystem path; reject a leading `-`, and the `ext::`/`fab::`/`file://-` transports. Keep the argv form `["remote", "add", "origin", remote_url]` (git's `remote add` does not take `--`), so validation is the defense.
  - **SYNC-11:** in `memory_ndjson::export`, **create the file with mode `0600` up front** rather than `fs::write`-then-chmod (which leaves a `0644` window). On `#[cfg(unix)]`, open via `tokio::fs::OpenOptions::new().write(true).create(true).truncate(true).mode(0o600).open(path)` (using `std::os::unix::fs::OpenOptionsExt`) and write the body to that handle, mirroring `secrets/file_store.rs:74-82`. On non-unix, fall back to `fs::write`.
- **Files touched** — `src/modules/sync/infrastructure/memory_ndjson.rs`, `src/modules/sync/domain/merge.rs`, `src/modules/sync/application/sync_service.rs`.
- **Tests first** — `import_rejects_oversized_file` (write a file over the byte cap → `Err`, nothing merged); `import_streams_within_caps` (round-trip still merges, regression over `export_then_import_round_trips`). `non_rfc3339_incoming_does_not_overwrite` (`incoming_wins("zzzz", "2026-…Z") == false`). `validate_remote_url_rejects_ext_and_dash` (`ext::sh -c x`, `-oProxyCommand=…` → `Err`) and `validate_remote_url_accepts_https_ssh_and_local` (→ `Ok`). `exported_ndjson_is_owner_only` (`#[cfg(unix)]`: assert mode `& 0o777 == 0o600` immediately after export — no transient-mode window).
- **ADR** — none (within ADR 0015's existing scope; note in its security section if convenient).
- **Risk & dependencies** — Low–Medium; independent of other steps.

### Step 8 — Scope semantic recall to the active embedding model (MEM-05)
- **Problem** — Every embedding is persisted with its `model` (SQLite also stores `dim`), but `embedded_candidates` selects only the vector and `semantic_pick` ranks the query against all stored vectors regardless of producing model. `cosine` returns `0.0` only on length mismatch, so two same-dimension models (e.g. both 1536-dim) are ranked on meaningless cross-model cosines that can clear `MIN_SIMILARITY` — changing the embedding model silently corrupts recall, though the guarding metadata is already on disk.
- **Approach** — Thread the active model id through the candidate fetch:
  - Change the port **trait-default** method `ProjectStore::embedded_candidates` (`project_store.rs:33`) and `SharedStore::embedded_candidates` (`shared_store.rs:37`) to `embedded_candidates(&self, model: &str, limit: usize)`. **Only the trait default signature changes** — `MockProjectStore`/`MockSharedStore` in `memory_port.rs` do **not** override this method (they rely on the default), so the test doubles need no edit.
  - `SqliteSharedMemory::embedded_candidates` (sqlite_shared_memory.rs:75): add `WHERE emb.model = ?1` and shift the `LIMIT` bind. `FileProjectMemory::embedded_candidates` (file_project_memory.rs:121): filter the sidecar to entries whose `StoredEmbedding.model == model` before sorting/truncating.
  - Update the delegating adapters (`file_project_store.rs:47-48`, `sqlite_shared_store.rs:50-51`) to forward `model`.
  - In `memory_port.rs::recall_project`/`recall_shared` (lines 145, 167), pass `embedder.model()` to `embedded_candidates(...)`. Drop the now-inaccurate "blunted by the floor" claim in the `MIN_SIMILARITY` doc-comment (the floor no longer carries that burden).

### Step 9 — Re-contain and de-symlink the memory/docs read paths; make the body write atomic (MEM-07, MEM-08, MEM-10)
- **Problem** — `load`/`search` join the stored `index_entry.path` onto `root` and read with no re-containment check (file_project_memory.rs:238,269), so a `../..` path from a corrupted/merged `index.json` reads outside the memory dir. `DocsLibrary::collect_markdown_files` descends with `path.is_dir()` (docs_library.rs:84,97, follows symlinks), so a symlink under `docs/` lets `consult_docs` surface files outside the docs root. - **Approach** —
  - **MEM-07:** add a small `fn contained_join(root: &Path, rel: &str) -> Option<PathBuf>` in `file_project_memory.rs` that rejects a `rel` containing a `ParentDir`/absolute component (lexical, mirroring `join_checked`) and returns `root.join(rel)` otherwise. `load` returns `Ok(None)` and `search` skips an entry whose stored path fails containment (defensive — never panics).

### Step 10 — Time the edit_file I/O and restore least-privilege for read-only tools (TOOL-06, TOOL-07)

### Step 11 — Constrain session message ordinals (SESS-08)
- **Problem** — The `messages` table already has a **non-unique** `CREATE INDEX IF NOT EXISTS idx_messages_session ON messages(session_id, ordinal)` (sqlite_session_store.rs:122) but no UNIQUE constraint, and `append_messages` computes `MAX(ordinal)+1` inside a DEFERRED transaction (`unchecked_transaction`, line 174), so two concurrent same-session writers can read the same MAX and insert duplicate ordinals (a corrupted replay order).
- **Approach** —
  - **Index (repurpose, do not duplicate):** in `init`, replace the existing non-unique index — `DROP INDEX IF EXISTS idx_messages_session; CREATE UNIQUE INDEX IF NOT EXISTS idx_messages_session_ordinal ON messages(session_id, ordinal);`. This avoids leaving two indexes on identical columns (redundant write cost). The unique index is the belt-and-suspenders that fails the insert if a race still produced a collision. (Open question: a legacy DB with pre-existing duplicate ordinals would fail the unique-index creation — documented as a known pre-1.0 migration caveat.)
  - **Transaction (keep RAII, switch to IMMEDIATE):** instead of `conn.unchecked_transaction()` (DEFERRED, whose `Drop` rolls back) followed by a raw `BEGIN IMMEDIATE` (which would have no RAII guard and could strand an open transaction on the shared `Arc<Mutex<Connection>>` on an early-return error), take the lock mutably and use the RAII `transaction_with_behavior`: `let mut guard = lock(&conn)?; let tx = guard.transaction_with_behavior(rusqlite::TransactionBehavior::Immediate).map_err(sess)?;` then the `MAX(ordinal)` read + inserts, then `tx.commit().map_err(sess)?;`. Any `?` early-return drops `tx`, which rolls back automatically — no stranded transaction. `MutexGuard<Connection>` derefs mutably, so `transaction_with_behavior(&mut self, …)` is callable.
- **Files touched** — `src/modules/session/infrastructure/sqlite_session_store.rs`.
- **Tests first** — `append_assigns_contiguous_ordinals` (regression: sequential appends get `0,1,2,…`). `duplicate_ordinal_is_rejected`: directly insert two rows with the same `(session_id, ordinal)` and assert the second errors (locks the unique index). `append_error_rolls_back`: force an insert failure mid-batch (e.g. a row that violates the unique index) and assert the transaction rolled back (no partial rows persisted, the connection is usable for the next write — covers the RAII rollback path). Concurrency is hard to force deterministically; the unique-index + rollback tests are the regression locks for the invariant.
- **ADR** — none (within ADR 0013's session scope).
- **Risk & dependencies** — Low; isolated to the session adapter.
