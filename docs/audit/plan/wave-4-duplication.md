# Wave 4 — Duplication consolidation — Remediation Plan

> Scope: single-source every block of logic that the multi-module audit found copied across `src/`, so it can drift no further — the protocol-critical `Role`↔wire mapping, the SQLite blocking-store harness, the provider send/stream/body-read skeleton (with Wave 1's byte cap landing once here), the `now_rfc3339`/`write_atomic` primitives, the `<E: Display>→AgentError` mappers, the discard/collect `EventSink` doubles, the tool-args sanitizer, the open-coded `Notice` constructor, the canonical `"function"` tool kind, and the `SYSTEM_PROMPT` values that re-type live tool/limit/sensitive-pattern facts. It ships as a unit because the fixes share their destinations (`shared/kernel`, `shared/infra`, `provider/infrastructure`) and several depend on each other (the typed `AgentError` constructors feed the shared SQLite harness; the kernel `"function"` const and the tool-args helper feed the shared provider loop). No behavior changes are intended except the security hardening that PROV-01's cap now also guards the Anthropic stream and that SEC-06 makes the prompt reflect the live sensitive matcher.

## Remediation steps (ordered to de-risk within the wave)

### Step 1 — Add typed `AgentError` constructors; delete the four `<E: Display>` mappers (DUP-07 / CONS-07)
- **Problem** — The same one-liner `fn name<E: Display>(error: E) -> AgentError { AgentError::Variant(error.to_string()) }` is hand-rolled four times under three names: `sqlite_shared_memory.rs::mem` (→ `Memory`), `file_project_memory.rs::mem` (→ `Memory`), `sqlite_session_store.rs::sess` (→ `Session`), and `sync/infrastructure/memory_ndjson.rs::ser` (→ **`Memory`**, because it (de)serializes memory entries even though it lives in the sync module). The kernel exposes no constructor, so every adapter writes its own and the "stringify into this variant" rule lives in four places. Maintainability; it also blocks Step 6 (the shared SQLite harness needs one canonical `fn(String)->AgentError` to parameterize on).
- **Approach** — In `src/shared/kernel/error.rs`, add an `impl AgentError` block with exactly the two constructors that have a real adopter: `pub fn memory(error: impl std::fmt::Display) -> Self { Self::Memory(error.to_string()) }` and the same for `session`. **Do not add `sync`/`provider`/`secret` constructors:** no bare `<E: Display>` mapper collapses onto them — every `AgentError::Sync`/`Provider`/`Secret` construction in the tree interpolates contextual text via `format!` (e.g. `sync/infrastructure/git_cli.rs:39`, `sync/application/sync_service.rs:90`), and Step 9 keeps building provider errors inline with context. Adding unused `pub fn`s in a single binary crate triggers `dead_code`, which `-D warnings` turns into a hard build failure — YAGNI plus the gate both forbid them. Replace the private mappers: `sqlite_shared_memory.rs::mem`, `file_project_memory.rs::mem`, and `memory_ndjson.rs::ser` → `AgentError::memory`; `sqlite_session_store.rs::sess` → `AgentError::session`. Standardize the still-needed closures on `.map_err(AgentError::memory)` / `.map_err(AgentError::session)` (function-as-mapper) where the source is `impl Display`, and on `|error| AgentError::memory(error)` only where extra context is interpolated; rename any stray `|e|` bindings to `|error|` in the touched files. Call the constructors by their fully-qualified `AgentError::memory`/`AgentError::session` form already imported in each file; no blanket re-export.
- **Files touched** — `src/shared/kernel/error.rs` (modified); `src/modules/memory/infrastructure/sqlite_shared_memory.rs`, `src/modules/memory/infrastructure/file_project_memory.rs`, `src/modules/session/infrastructure/sqlite_session_store.rs`, `src/modules/sync/infrastructure/memory_ndjson.rs` (modified).
- **Tests first** — In `error.rs` `#[cfg(test)]`: `memory_constructor_builds_memory_variant`, `session_constructor_builds_session_variant` (each asserts the `Display` text round-trips into the variant's message); `constructor_accepts_any_display_source` (passes a `&str`, an `io::Error`, and a `rusqlite`-style error stand-in). Regression: the existing memory/session store suites and the `memory_ndjson` round-trip tests must stay green unchanged (the variant each error maps to is identical to today — `ser` still maps to `Memory`).
- **ADR** — none.
- **Risk & dependencies** — Blast radius tiny (mechanical rename). No wave dependency; **unblocks Step 6**.

### Step 2 — Single-source the `Role`↔wire-string mapping on the kernel (DUP-04 / SESS-02)
- **Problem** — The four-arm `Role`→`"system"/"user"/"assistant"/"tool"` mapping exists in three independent copies: `openai/message_dto.rs::wire_role`, `session/message_dto.rs::role_to_str`, and the inverse `role_from_str`. The session store persists these strings to SQLite and the provider sends them on the wire; if one copy ever renames a variant's spelling and another does not, stored and sent history diverge silently — the single most dangerous drift in the tree.
- **Approach** — On `src/shared/kernel/role.rs`, add `pub const fn as_wire_str(self) -> &'static str` (the four canonical lowercase arms) and `pub fn from_wire_str(s: &str) -> Option<Role>` (the inverse, `None` for unknown so a corrupt row is skipped defensively). Keep the enum serde-free — these are inherent methods, **not** a `Serialize`/`Deserialize` impl. Update the `Role` doc comment to point at `as_wire_str`/`from_wire_str` as the single source. Then: in `openai/message_dto.rs`, delete `wire_role` and have `serialize_role` call `role.as_wire_str()`; in `session/message_dto.rs`, delete `role_to_str`/`role_from_str` and call `Role::as_wire_str`/`Role::from_wire_str` from `From<&Message>` and `into_domain`.
- **Files touched** — `src/shared/kernel/role.rs` (modified); `src/modules/provider/infrastructure/openai/message_dto.rs`, `src/modules/session/infrastructure/message_dto.rs` (modified).
- **Tests first** — In `role.rs`: `as_wire_str_maps_all_four_variants`; `from_wire_str_round_trips_all_four`; `from_wire_str_rejects_unknown_role` (`"bogus" → None`); `wire_round_trip_is_identity` (for each variant, `from_wire_str(v.as_wire_str()) == Some(v)`). Keep the provider DTO's `wire_role_maps_all_variants` test but re-point it at `as_wire_str`. Regression: the session DTO's `round_trips_an_assistant_tool_call`, `round_trips_a_tool_result`, and `unknown_role_is_skipped` must stay green (now exercising the kernel methods).
- **ADR** — none (preserves the serde-free-domain decision in ADR 0003; record nothing new).
- **Risk & dependencies** — Protocol-critical but compiler-checked (a missed arm fails to build). No wave dependency.

### Step 5 — One `write_atomic` in `shared/infra/fs.rs` (DUP-06 / SYNC-05)
- **Problem** — Two private `write_atomic` helpers implement the same write-temp-then-rename crash-safety idiom with the same motivating comment, differing only in temp-name scheme (`with_extension("json.tmp")` in `file_project_memory.rs` vs `.{name}.kiri-tmp` in `sync_service.rs:239`). A hardening to one (e.g. fsync-before-rename) would not reach the other.
- **Approach** — Add `src/shared/infra/fs.rs` with `pub async fn write_atomic(path: &Path, content: &[u8]) -> Result<(), AgentError>`: derive the temp sibling with the **canonical** policy `.{file_name}.kiri-tmp` (sync's scheme — works for any filename, including those that already carry an extension, unlike memory's `.json.tmp`), `tokio::fs::write` the temp then `rename` over `path`; the `io::Error`s flow to `AgentError::Io` via the existing `#[from]`. Register `pub mod fs;` in `src/shared/infra.rs`. Delete both local helpers: `file_project_memory.rs` (infrastructure) calls the shared one (passing `content.as_bytes()`), and `sync_service.rs::write_atomic` is removed, its callers using the shared one. Document the temp-name policy in one place.
- **Boundary note (gap acknowledgement)** — `sync_service.rs` lives in the sync **application** layer and already performs `tokio::fs::write` there today (a pre-existing application-layer I/O); after the move it calls `shared::infra::fs::write_atomic` (application → shared/infra). This is the **same consumption pattern as `shared/infra/config`**, which is already read across layers, and the data-dir-ownership exemption for `sync` covers it. The move does **not worsen** the boundary — it removes a duplicated raw `fs::write` and routes it through one audited primitive. The memory caller is infrastructure, which is clean. No new ADR is needed; this is called out explicitly rather than implying both callers are infrastructure.
- **Files touched** — `src/shared/infra/fs.rs` (new); `src/shared/infra.rs` (modified); `src/modules/memory/infrastructure/file_project_memory.rs`, `src/modules/sync/application/sync_service.rs` (modified).
- **Tests first** — `fs.rs` (tempdir): `write_atomic_creates_file_with_content`; `write_atomic_overwrites_existing_file`; `write_atomic_leaves_no_temp_sibling_on_success`; `write_atomic_handles_a_name_with_an_existing_extension` (e.g. `config.toml` → temp `.config.toml.kiri-tmp`, target written). Regression: the memory store's atomic index/embeddings tests and the sync config-write tests (no `.tmp` residue, content intact) stay green.
- **ADR** — none.
- **Risk & dependencies** — Low; the temp-name change for memory (`.json.tmp` → `.{name}.kiri-tmp`) is internal and invisible to readers. No wave dependency.

### Step 6 — Extract the SQLite blocking-store harness to `shared/infra/sqlite.rs` (DUP-01 / SESS-01)
- **Problem** — `memory` and `session` reimplement the same blocking-connection harness: `lock(conn)` mapping a poisoned mutex, `const DB_OP_TIMEOUT = 5s`, `async fn run_blocking` wrapping `spawn_blocking` in a `tokio::time::timeout`, and `new(db_path)` doing `create_dir_all(parent)` then `Connection::open`. The only differences are the `AgentError` variant and the session store's extra `busy_timeout`/WAL pragmas. This is the largest cross-module copy; a change to the timeout, the poison message, or the join-error handling must be made twice.
- **Approach** — Add `src/shared/infra/sqlite.rs`, parameterized by an error constructor `map_err: fn(String) -> AgentError` (Step 1's `AgentError::memory`/`AgentError::session` fit exactly — a generic inherent fn instantiated at `String` coerces to the `fn(String) -> AgentError` pointer; this is valid Rust):
  - `pub const DB_OP_TIMEOUT: Duration = Duration::from_secs(5);`
  - `pub fn open_with_parent(db_path: &Path, map_err: fn(String) -> AgentError) -> Result<Connection, AgentError>` (creates the parent dir, `Connection::open`, classifies the open error through `map_err`).
  - `pub fn lock<'a>(conn: &'a Mutex<Connection>, map_err: fn(String) -> AgentError) -> Result<MutexGuard<'a, Connection>, AgentError>`.
  - **A testable timeout seam:** `pub async fn run_blocking_with_timeout<T: Send + 'static>(op: impl FnOnce() -> Result<T, AgentError> + Send + 'static, timeout: Duration, map_err: fn(String) -> AgentError) -> Result<T, AgentError>` owns the `spawn_blocking` + `tokio::time::timeout` + join/timeout mapping. The join-error branch builds its message via `map_err(join_error.to_string())` (the ctor takes `String`, so the `JoinError` must be stringified first); the timeout branch builds `map_err("database operation timed out".to_string())`. Then `pub async fn run_blocking<T: Send + 'static>(op, map_err) -> Result<T, AgentError>` simply delegates: `run_blocking_with_timeout(op, DB_OP_TIMEOUT, map_err).await`. Production code calls `run_blocking`; the timeout test points at `run_blocking_with_timeout` with a tiny `Duration` so it never sleeps 5s.
  - Register `pub mod sqlite;` in `src/shared/infra.rs`.
  - Each store deletes its private `lock`/`DB_OP_TIMEOUT`/`run_blocking`, keeps only its schema/queries, and threads its `AgentError::memory`/`AgentError::session`. The session store keeps its `busy_timeout(DB_OP_TIMEOUT)` + WAL `pragma_update` calls (referencing the shared const) immediately after `open_with_parent`; the memory store does not. `in_memory()`/`in_memory_inert()` stay per-store (the inert-flag modeling difference is SESS-07, out of this wave).
- **Files touched** — `src/shared/infra/sqlite.rs` (new); `src/shared/infra.rs` (modified); `src/modules/memory/infrastructure/sqlite_shared_memory.rs`, `src/modules/session/infrastructure/sqlite_session_store.rs` (modified).
- **Tests first** — `sqlite.rs`: `run_blocking_returns_the_op_result`; `run_blocking_maps_a_panicking_closure_via_the_constructor` (a closure that panics → `Err` built by the passed ctor, asserting the variant); `run_blocking_times_out_via_the_constructor` (drive `run_blocking_with_timeout` with a near-zero `Duration` and a closure that blocks past it → the ctor's "timed out" message — **the seam is what makes this test possible without a real 5s sleep**); `lock_maps_a_poisoned_mutex_via_the_constructor`; `open_with_parent_creates_the_missing_parent_dir`. Regression: both stores' full existing suites (memory save/search/embedded_candidates, session create/append-order/list/latest/delete/reopen/inert) must stay green, proving the harness swap is behavior-preserving and each store still emits its own variant.
- **ADR** — none (`shared/infra/sqlite.rs` sits beside the existing `shared/infra/config`, consistent with harness-owned-I/O placement).
- **Risk & dependencies** — Medium blast radius (two persistence adapters), fully covered by their existing suites. **Depends on Step 1** (the constructors). **Depends on Step 4** (both stores call `now_rfc3339`).

### Step 8 — One `tool_args` sanitizer for the provider adapters (DUP-13)
- **Problem** — Three functions encode the same "trim; empty-or-non-JSON falls back to `{}` so a garbled turn can never poison a later request": `openai/arguments.rs::normalize_arguments` (String, escapes first), `anthropic/sse.rs::tool_input_to_arguments` (String), `anthropic/message_dto.rs::parse_input` (Value). The validation rule and its rationale comment are written three times.
- **Approach** — Add `src/modules/provider/infrastructure/tool_args.rs` with the single owner of the rule:
  - `pub(crate) fn sanitized_object(raw: &str) -> serde_json::Value` (trim; empty or non-JSON → `json!({})`; else the parsed value).
  - `pub(crate) fn sanitized_string(raw: &str) -> String` — **trim only for the parse check, but return the ORIGINAL `raw` (untrimmed) on success**, and `"{}"` otherwise. This exactly preserves today's `tool_input_to_arguments`, which returns the untrimmed `input` when `serde_json::from_str(trimmed).is_ok()` (a behavior change here would alter the bytes re-sent to the provider).
  - Register `pub(crate) mod tool_args;` in `provider/infrastructure.rs`. Repoint: `anthropic/message_dto.rs::parse_input` → `sanitized_object`; `anthropic/sse.rs::tool_input_to_arguments` → `sanitized_string`. **Preserve OpenAI's distinct escaping:** `openai/arguments.rs::normalize_arguments` keeps `escape_to_owned` (its unique control-char fix) then delegates the validate-or-`{}` decision to `tool_args::sanitized_string(&escaped)` — so the shared rule is single-sourced while the escaping stays local to OpenAI, and `sanitized_string` returning its `raw` argument means OpenAI gets back its escaped string unchanged.
- **Files touched** — `src/modules/provider/infrastructure/tool_args.rs` (new); `src/modules/provider/infrastructure.rs` (modified); `src/modules/provider/infrastructure/openai/arguments.rs`, `src/modules/provider/infrastructure/anthropic/sse.rs`, `src/modules/provider/infrastructure/anthropic/message_dto.rs` (modified).
- **Tests first** — `tool_args.rs`: `sanitized_object_empty_input_is_empty_object`; `sanitized_object_non_json_is_empty_object`; `sanitized_object_parses_valid_json`; `sanitized_string_keeps_valid_json_untrimmed` (assert leading/trailing whitespace around valid JSON is preserved in the return, locking the no-behavior-change rule); `sanitized_string_falls_back_for_garbled`; `sanitized_string_empty_is_brace_brace`. Regression (load-bearing): OpenAI's `normalizes_raw_control_chars_in_tool_call_arguments` must stay green (escaping still runs before the shared validation); Anthropic's `tool_use_with_no_input_defaults_to_empty_object` and `truncated_tool_input_falls_back_to_empty_object` stay green.

### Step 9 — Extract the shared provider send/stream/body-read path; land PROV-01's byte cap once (DUP-02 / PROV-02 / DUP-03 / PROV-01)
- **Problem** — All three HTTP adapters repeat the non-success preamble verbatim (`status` → on failure `response.text().await.unwrap_or_else(|error| "<error body unavailable: {error}>")` → `error_from_status`). The two chat adapters additionally repeat the `bytes_stream().eventsource()` + `tokio::pin!` + `while let Some(event)` + read-error mapping + `handle_event` loop, and carry parallel `TurnAccumulator`s whose `hit_empty_output_limit` differs only by sentinel. Critically, the OpenAI accumulator bounds `content + tool-arg` bytes by `MAX_STREAM_BYTES` against an untrusted/unbounded stream (`openai/sse.rs:17,55-58`, the accumulator carries a `streamed_bytes` field at `sse.rs:133`), but **the Anthropic accumulator has no equivalent ceiling** — its `TurnAccumulator` (`anthropic/sse.rs:81-87`) has no `streamed_bytes` field at all (PROV-01) — the identical OOM threat is unguarded for one adapter.
- **Approach** — Add `src/modules/provider/infrastructure/streaming.rs` (beside `http_error`) owning:
  - `pub(crate) async fn read_error_body(response: reqwest::Response) -> String` — the `<error body unavailable: {error}>` reader, shared by all three.
  - `pub(crate) async fn ensure_success(response: reqwest::Response) -> Result<reqwest::Response, AgentError>` — `if !status.is_success() { return Err(error_from_status(status, read_error_body(response).await)); }` else `Ok(response)`. All three adapters call it, replacing their copied preamble.
  - `pub(crate) async fn drain_sse(response: reqwest::Response, mut on_data: impl FnMut(&str) -> Result<(), AgentError>) -> Result<(), AgentError>` — owns `bytes_stream().eventsource()` + `tokio::pin!` + the `while let Some(event)` loop and the `error reading stream: {error}` mapping. Each chat adapter calls `drain_sse(response, |data| handle_event(data, &mut accumulator, sink)).await?;`, so the per-provider ingestion (`handle_event` + the provider's `TurnAccumulator`) stays in its module while the loop exists once.
  - `pub(crate) const MAX_STREAM_BYTES: usize = 8 * 1024 * 1024;` and `pub(crate) fn enforce_stream_budget(running: &mut usize, delta_bytes: usize) -> Result<(), AgentError>` (saturating add; error past the cap with the existing `provider stream exceeded the maximum response size ({MAX_STREAM_BYTES} bytes)` message). **PROV-01 lands here for both adapters:**
    - OpenAI moves its existing inline check (`openai/sse.rs:55-58`) to call `enforce_stream_budget(&mut accumulator.streamed_bytes, delta_bytes)`; its accumulator keeps the `streamed_bytes: usize` field.
    - **Anthropic gains it for the first time:** add a `streamed_bytes: usize` field to `anthropic/sse.rs::TurnAccumulator` (the `#[derive(Default)]` struct at lines 81-87), and call `enforce_stream_budget(&mut self.streamed_bytes, delta_bytes)` **before absorbing** in both ingestion paths — in `apply_delta` for the `TextDelta` branch (`delta_bytes = text.len()`) and in `push_tool_input` for the `InputJsonDelta` branch (`delta_bytes = partial.len()`) — mirroring OpenAI's `handle_event` placement. If Wave 1 has already added an inline cap to `anthropic/sse.rs`, replace it with this shared call so no duplicate ceiling remains.
  - `pub(crate) fn is_empty_truncation(reason_is_cap: bool, content: &str, tool_calls_empty: bool) -> bool { reason_is_cap && content.is_empty() && tool_calls_empty }` — the shared empty-output predicate both `hit_empty_output_limit`s delegate to (each passing its own sentinel comparison via `reason_is_cap`). Keep the per-provider `TurnAccumulator` structs and their distinct ingestion (DUP-03 stops here — a shared index-keyed builder is not extracted, to avoid speculative abstraction over two genuinely different fragment shapes; see open question).
  - Register `pub(crate) mod streaming;` in `provider/infrastructure.rs`.
- **Files touched** — `src/modules/provider/infrastructure/streaming.rs` (new); `src/modules/provider/infrastructure.rs` (modified); `src/modules/provider/infrastructure/openai/provider.rs`, `src/modules/provider/infrastructure/anthropic/provider.rs`, `src/modules/provider/infrastructure/openai/embeddings.rs`, `src/modules/provider/infrastructure/openai/sse.rs`, `src/modules/provider/infrastructure/anthropic/sse.rs` (modified).
- **Tests first** — `streaming.rs` (hermetic loopback servers, mirroring the existing provider tests): `ensure_success_passes_through_a_2xx`; `ensure_success_classifies_a_4xx_with_its_body` and `..._a_5xx`; `read_error_body_reports_unavailable_when_the_body_read_fails`; `drain_sse_feeds_each_data_payload_to_the_handler`; `drain_sse_maps_a_read_error_to_provider`; `enforce_stream_budget_allows_within_the_cap`, `..._errors_past_the_cap`, `..._saturates_without_overflow`; `is_empty_truncation_truth_table`. **PROV-01 regression lock (the security fix):** `anthropic_stream_exceeding_the_cap_fails_fast` — drive the Anthropic accumulator (via repeated `TextDelta`/`InputJsonDelta` through `handle_event`) past `MAX_STREAM_BYTES` and assert the provider error, the lock that Anthropic is now guarded. Regression: every existing OpenAI/Anthropic accumulator, in-band-error, truncation, and loopback timeout test stays green.
- **ADR** — none (the cap policy is unchanged; it is merely shared and extended — record the Anthropic coverage in the PROV-01 follow-up, not a new ADR).
- **Risk & dependencies** — Highest blast radius in the wave (both streaming adapters + embeddings), but each touched path has strong existing coverage. **Depends on Step 3** (kernel `"function"` const), **Step 7** (`NullSink`/`CollectSink`), **Step 8** (`tool_args`). **Cross-wave:** coordinate with Wave 1's PROV-01 so the cap is not double-applied.

### Step 10 — `Model::notify*` + a `persist_or_notice` helper (TUII-02)
- **Problem** — `…push(TranscriptItem::Notice(NoticeLevel::Info|Error, format!(...)))` is open-coded tree-wide: **47 sites in `runtime.rs`, 10 in `keymap.rs`** (432, 443, 454, 466, 495, 535, 628, 633, 687, 793), **1 in `update.rs:48`, and 1 self-site in `model.rs:190`** (`enter_onboarding`). The audit's "49 sites" is tree-wide, not runtime-only; scoping the fix to `runtime.rs` would leave ~12 production sites un-collapsed and miss the single-source goal. Separately, the "persist failed → push error notice" block recurs across `apply_set_model`/`apply_set_effort`/`apply_set_provider`/`apply_save_provider`.
- **Approach** — On `Model` in `src/modules/tui/domain/model.rs`, add a base method and two level-fixed wrappers (the base also serves the one dynamic-level site):
  - `pub fn notify(&mut self, level: NoticeLevel, message: impl Into<String>) { self.transcript.push(TranscriptItem::Notice(level, message.into())); }`
  - `pub fn notify_info(&mut self, message: impl Into<String>) { self.notify(NoticeLevel::Info, message); }`
  - `pub fn notify_error(&mut self, message: impl Into<String>) { self.notify(NoticeLevel::Error, message); }`
  - Rewrite **all construction sites** through these: in `runtime.rs` the 47 `model.notify_info(...)`/`model.notify_error(...)` (and `transcript.push` variants like the `transcript.push` at `runtime.rs:1337` via the surrounding `&mut Model`); in `keymap.rs` the 10 sites — the level-fixed ones use `model.notify_info`/`notify_error`, and **`keymap.rs:628`, which pushes with a dynamic `level` variable, uses `model.notify(level, label)`** (this is exactly why the base `notify(level, …)` exists); in `update.rs:48` `model.notify_info(label)`; and the self-site in `model.rs:190` `self.notify_info(ONBOARDING_WELCOME)`. These are all `application`/`domain`/`infrastructure` callers with a `&mut Model` (or `self`) in scope, and application/infrastructure calling a domain method is an inward dependency — architecture-clean.
  - For the persist-then-report pattern, add a free helper in `runtime.rs` (runtime glue, not domain): `fn persist_or_notice<E: std::fmt::Display>(result: Result<(), E>, model: &mut Model, context: &str)` that, on `Err(error)`, calls `model.notify_error(format!("{context}: {error:#}"))` and on `Ok` does nothing. Generic over `E: Display` so it survives SHARED-02 changing the config writers from `anyhow::Result` to `Result<_, AgentError>` (today `config::persist_*` return `anyhow::Result<()>`, whose `anyhow::Error: Display`). Route the four persist sites through it:
    - The three `if let Err(error) = config::persist_*` blocks (`persist_active_model`, `persist_effort`, `persist_active_provider`) become `persist_or_notice(config::persist_*(…), model, "…")`.
    - The **`.and_then` chain at `runtime.rs:1127-1128`** (`config::upsert_provider(...).and_then(|()| config::persist_active_provider(...))`) is consumed by binding the composed result first, then passing it: `let persisted = config::upsert_provider(config_path, &profile).and_then(|()| config::persist_active_provider(config_path, &id)); persist_or_notice(persisted, model, "provider ativo, mas não persistiu no config");` — the chained semantics are unchanged; only the report path is shared.
- **Files touched** — `src/modules/tui/domain/model.rs` (modified); `src/modules/tui/infrastructure/runtime.rs`, `src/modules/tui/application/keymap.rs`, `src/modules/tui/application/update.rs` (modified).
- **Tests first** — `model.rs`: `notify_info_pushes_an_info_notice`; `notify_error_pushes_an_error_notice`; `notify_pushes_with_the_given_level` (assert the trailing `TranscriptItem::Notice` level + text for a dynamic level). In `runtime.rs` `#[cfg(test)]`: `persist_or_notice_is_silent_on_ok`; `persist_or_notice_pushes_an_error_notice_on_err` (the regression lock that a persist failure is still surfaced, never swallowed — the contract's "no silent no-ops"). Regression: any existing test asserting a `/models` / `/effort` / `/provider` failure surfaces a notice, and the `keymap` auto-mode notice test (`matches!(last, …"modo auto ativo")`) stay green.
- **ADR** — none.
- **Risk & dependencies** — Mechanical but wide; the helper preserves the exact `{error:#}` rendering and `notify` preserves the exact level/text. No wave dependency (independent of the TUII-01 god-file split — these helpers move cleanly into whatever submodule that split produces).

### Step 11 — Render the system prompt's tool/limit/sensitive facts from live single sources (SHARED-04 / SEC-06)
- **Problem** — `SYSTEM_PROMPT` prose restates values owned elsewhere and silently drifts: the sensitive-filename list (`config.rs:108-111` duplicates `DEFAULT_SENSITIVE_PATTERNS`), the run_command limits "30s timeout … 64 KiB" (`config.rs:61` duplicates the tool's own constants), "Every ~30 minutes" (`config.rs:92` duplicates `TOOL_CHECKPOINT`), and "ten file tools" (`config.rs:44` duplicates `default_fs_tools`). The security-critical one (SEC-06): `KIRI_SENSITIVE_PATTERNS` can override the live matcher while the prompt keeps asserting the hardcoded defaults — so **the prompt lies to the model about what is actually blocked**. The limits are doubly-duplicated even within the tool: the enforced 64 KiB output cap already has a named single source `EXEC_MAX_BYTES` at `tools/infrastructure/exec.rs:17` (used by `exec::capped_combined`), and the enforced default timeout already has a single source `tools/infrastructure/args.rs:60 default_timeout_ms() -> 30_000` (the `#[serde(default)]` on `RunCommandArgs.timeout_ms`) — yet `30000` is *also* hardcoded in the run_command JSON schema literal (`run_command.rs:122`), its description (`run_command.rs:104,121`), and the prompt (`config.rs:61`), and `64 KiB` is hardcoded in the description (`run_command.rs:104`) and the prompt.
- **Approach** — Make every dynamic fragment derive from its **existing** single source; introduce no second copy of any value:
  - **Output cap (64 KiB): reuse `EXEC_MAX_BYTES`.** Do **not** create a new `RUN_COMMAND_OUTPUT_CAP_BYTES` — that would be a second source of truth for the same 64 KiB that enforcement (`exec::capped_combined`) does not read. The run_command schema description and the prompt both render their `… KiB` fragment from `exec::EXEC_MAX_BYTES` (`run_command.rs` already imports `exec`).
  - **Default timeout (30000): one const reached by all four consumers.** Add `pub const RUN_COMMAND_DEFAULT_TIMEOUT_MS: u64 = 30_000;` in `tools/infrastructure/args.rs` (the lowest-level module — `run_command.rs` already depends on `args.rs`, so placing it there avoids a module cycle), and route **all four** sites through it: (1) `args.rs::default_timeout_ms()` returns `RUN_COMMAND_DEFAULT_TIMEOUT_MS` (the enforced default); (2) the run_command schema's `json!` literal becomes `"default": RUN_COMMAND_DEFAULT_TIMEOUT_MS` (the `json!` macro interpolates the const); (3) the run_command description; (4) `app.rs` passes it to the prompt renderer. The enforced default and every advertised default now read one value — closing the SEC-06-style "advertised vs enforced" gap for the timeout.
  - **Schema description becomes a `format!`.** The run_command description at `run_command.rs:99-128` is currently a concat-style `&str` literal; naming a const does not by itself make it read the const. Convert the description argument of `function_schema(self.name(), …, …)` to `&format!("Run a shell command … Output is truncated at {} KiB. A timeout (default {} ms / {}s) prevents runaway commands.", exec::EXEC_MAX_BYTES / 1024, RUN_COMMAND_DEFAULT_TIMEOUT_MS, RUN_COMMAND_DEFAULT_TIMEOUT_MS / 1000)` (`function_schema` takes `description: &str`). The schema the model sees and the prompt read the same single source.
  - **Sensitive list (SEC-06): render from the live matcher.** Add `SensitiveMatcher::globs(&self) -> Vec<&str>` in `tools/infrastructure/sensitive.rs` (returns the original globs the matcher was built from — the `(String, Regex)` pairs already store them).
  - **Checkpoint:** keep `TOOL_CHECKPOINT` as the source; the renderer receives the **effective** `settings.checkpoint_budget` (which defaults to `TOOL_CHECKPOINT` but is overridable) and renders `~{} minutes`.
  - **Tool count:** drop the brittle word "ten" in favor of "the file tools listed below" (the enumerated list immediately follows, so no count is asserted).
  - In `config.rs`, replace the fully-static `SYSTEM_PROMPT` const with a `SYSTEM_PROMPT_TEMPLATE` (the static prose) plus `pub fn render_system_prompt(sensitive_globs: &[&str], default_timeout_ms: u64, output_cap_bytes: usize, checkpoint: Duration) -> String` that interpolates the four fragments: the Security section's sensitive list = the joined `sensitive_globs`; the run_command line's limits = `default_timeout_ms`/1000 → seconds and `output_cap_bytes`/1024 → KiB; the Turn-mechanics checkpoint = `checkpoint.as_secs() / 60` minutes. `render_system_prompt` takes the values as parameters so `config.rs` gains **no** new dependency on `tools/infrastructure` (and does not deepen SHARED-01).
  - Remove `Settings.system_prompt: &'static str` (`config.rs:606,716`); in `app::wire`, replace `settings.system_prompt.to_string()` (`app.rs:143-147`) with `config::render_system_prompt(&settings.sensitive.globs(), tools::…::args::RUN_COMMAND_DEFAULT_TIMEOUT_MS, tools::…::exec::EXEC_MAX_BYTES, settings.checkpoint_budget)` and feed the result into the existing memory-digest concatenation. `app.rs` is the composition root and can reach both consts and the live `settings.sensitive` matcher (it is `.clone()`d into the sandbox at `app.rs:74`, so it remains readable here).
- **Files touched** — `src/modules/tools/infrastructure/sensitive.rs` (modified — add `globs`); `src/modules/tools/infrastructure/args.rs` (modified — add `RUN_COMMAND_DEFAULT_TIMEOUT_MS`, `default_timeout_ms` returns it); `src/modules/tools/infrastructure/fs/run_command.rs` (modified — description `format!` reading both consts; `json!` `default` reads the const); `src/shared/infra/config.rs` (modified — template + `render_system_prompt`, drop the `system_prompt` field); `src/app.rs` (modified — call the renderer).
- **Tests first** — `config.rs`: `system_prompt_lists_the_active_sensitive_globs` (render with a custom glob set, assert the prose contains exactly those, not the defaults); **`system_prompt_reflects_a_sensitive_override` (the SEC-06 security lock** — render with an override-derived glob list and assert the Security section equals the override, proving the prompt cannot lie); `system_prompt_renders_limits_from_the_named_constants` (assert the run_command line carries the `EXEC_MAX_BYTES`-derived KiB and `RUN_COMMAND_DEFAULT_TIMEOUT_MS`-derived seconds, so changing a const changes the text); `system_prompt_renders_the_checkpoint_minutes_from_the_budget`; `system_prompt_keeps_all_nine_section_headers` (regression that no prose was lost in the static→template move). `args.rs`: `default_timeout_ms_equals_the_const`. `run_command.rs`: `schema_default_timeout_equals_the_const` (assert the `json!` `default` field equals `RUN_COMMAND_DEFAULT_TIMEOUT_MS`); `schema_description_reflects_the_limit_constants`. `sensitive.rs`: `globs_returns_the_original_patterns`.
- **ADR** — Update `docs/decisions/0007-system-prompt-revision.md`: record that the tool limits, checkpoint interval, and sensitive-pattern fragment are rendered from live single sources (`EXEC_MAX_BYTES`, `RUN_COMMAND_DEFAULT_TIMEOUT_MS`, `TOOL_CHECKPOINT`, and the active `SensitiveMatcher`) rather than hardcoded, so a `KIRI_SENSITIVE_PATTERNS` override is reflected and the prompt cannot drift from what the harness enforces.
- **Risk & dependencies** — Medium; changes the prompt from a `&'static str` to a rendered `String`. Independent of the other steps. **Cross-wave:** if the architecture wave splits `config.rs` (SHARED-03), `render_system_prompt` + the template belong in that submodule; the renderer's signature is split-agnostic.

## Within-wave sequencing
1. **Step 1 (typed `AgentError` constructors)** first — it is the `fn(String)->AgentError` primitive the shared SQLite harness (Step 6) is parameterized on, and it touches the kernel error type every later step compiles against.
2. **Step 2 (Role wire methods)** — independent, protocol-critical, kernel-local; landing it early de-risks any later provider/session touch.
3. **Step 3 (kernel `"function"` const + sentinels)** — a kernel/provider primitive Step 9's shared accumulator code references; cheap, do it before the provider consolidation.
4. **Step 4 (shared `now_rfc3339`)** and **Step 5 (shared `write_atomic`)** — small leaf primitives the persistence adapters consume; land before Step 6 so the SQLite store touch is a single coherent edit.
5. **Step 6 (shared SQLite harness)** — depends on Step 1 (the constructors) and Step 4 (the timestamp); the largest persistence change, sequenced after its primitives exist.
6. **Step 7 (`NullSink`/`CollectSink`)** and **Step 8 (`tool_args`)** — the two provider-side primitives Step 9 reuses; land them before the big provider extraction so Step 9 is purely structural.
7. **Step 9 (shared provider stream path + PROV-01 cap)** — the highest-blast-radius step, sequenced last among the provider work so it composes Steps 3/7/8 and is the single place Wave 1's cap lands.
8. **Step 10 (TUI `notify*`)** and **Step 11 (system-prompt render)** — independent surface areas (TUI, config) with no dependency on the kernel/provider/persistence steps; either can go last. Step 11 carries the wave's remaining security lock (SEC-06).

This order means every consumer is edited only after its single-sourced primitive exists, so no step temporarily duplicates what a later step removes.
