Trigger on any task spanning multiple files, a new feature, a migration, or an ambiguous ask; skip it
for a one-line edit.

Read and trace the real flow first — the files the change touches, the existing patterns — before
writing anything down. Then write a short numbered plan: the goal, the files to touch, the order, and
the unknowns or risks.

When the task is ambiguous or destructive, confirm the plan with the user before editing.

Keep it the smallest plan that covers the change; revise it as you learn more instead of planning every
detail up front.
